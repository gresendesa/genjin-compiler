## Clarificações de linguagem obtidas no planning

Respostas do PO que determinam o escopo da documentação:

| Ponto | Definição |
|---|---|
| Herança de variável | `exec` sem `>>` herda a variável do bloco pai. Qualquer `exec` pode redeclarar com `>>`. |
| Posição do `while` | Sempre fecha o exec imediatamente acima, após o `}`. |
| Blocos raiz | Exatamente um bloco exec raiz por programa. |
| Formato do `from` | String com caminho. Split no último ponto: antes = biblioteca, depois = nome da macro. Ex: `from "Net.check"` → `ATTRIBUTE.MACRO: ('Net', 'check')`. |
| Comentários | Suportados: `//` (linha) e `/* */` (bloco). |
| `pass` | Sempre obrigatório no corpo de um exec. Lista os códigos não tratados localmente. |
| `case` | Sempre leva a um exec filho — nunca é folha. |
| Nome do bloco | Padrão = nome do proc invocado. Pode ser redeclarado com `as "nome"`. Ex: `exec esperar(segundos=5) as "wait" { ... }`. |
| `description` | Não existe na sintaxe `.gnj`. |
| Corpo do exec | Nunca vazio — deve ter ao menos um `case` ou `pass`. |
