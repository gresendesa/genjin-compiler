## Objetivo

Avaliar a viabilidade e a melhor forma de suportar importação de proc-blocos definidos em outros arquivos `.gnj`, permitindo reutilização de lógica entre programas Genjin distintos. Este item é exclusivamente de estudo — nenhuma implementação é entregue aqui.

---

## Motivação

Com B-017/B-019/B-020, proc-blocos permitem reutilização de lógica **dentro** de um mesmo arquivo `.gnj`. O próximo passo natural para completar a linguagem é permitir que proc-blocos sejam importados de arquivos externos — comportamento análogo ao `import` de linguagens convencionais.

Este recurso é considerado o último toque necessário para lançar a v1.0 oficial da linguagem.

---

## Contexto arquitetural relevante

### Dois sistemas de path distintos (CRÍTICO)

| Sistema | Resolve em | Usado para | Exemplo |
|---|---|---|---|
| Template path (Jinja2) | Fase assembler (`assembler.py`) | `from "X.Y.Z"` em declaração de proc normal | `"Federal.@.server_tools.serviço_teleportar"` |
| Código source path (.gnj) | Fase compilador (scanner/parser/desugar) | `from "dir1.dir2.dirn" import ...` (nova feature) | `"Macrosoft.common.ferramentas"` |

A separação é essencial: a resolução de templates ocorre em runtime (assembler), enquanto a resolução de código fonte deve ocorrer em compile-time. Misturá-los seria um erro arquitetural.

### Estado atual do scanner

O scanner já reconhece `KW_FROM` como keyword. **Não existe** `KW_IMPORT`.

### Estado atual do parser

Em `_parse_procs`, a detecção de uma declaração começa pelo IDENT (nome do proc). A linha de detecção está em:

```python
# parser.py, linha ~513
if self._check(TokenType.KW_FROM):
    # Proc normal: NomeProcedimento(...) from "path" { codes ... }
```

Esse `KW_FROM` pertence à cauda de `NomeProcedimento(...) from "..."`, não ao início da linha. Um `from "..." import ...` teria `KW_FROM` como **primeiro token** dentro do bloco `procs {}`, antes de qualquer IDENT — portanto não há ambiguidade gramatical com a sintaxe existente.

## Sintaxe proposta pelo PO

```gnj
procs {
    // declarações normais...

    from "dir1.dir2.dirn" import
        Exceção_e_Reinicia,
        Troca_Ferramenta
}
```

Onde `"dir1.dir2.dirn"` é um caminho dotted que resolve para o arquivo `dir1/dir2/dirn.gnj` relativo a uma raiz configurável (candidata: diretório do arquivo compilado, ou um path base configurável via CLI).

## Questões a responder no estudo

### Q1 — A sintaxe proposta é gramaticalmente coerente?

**Análise:** A sintaxe `from "..." import Name1, Name2` com `import` como palavra-chave é consistente com Python, familiar para leitores e não conflita com a sintaxe `from "..."` dos procs normais (que sempre vêm após um IDENT). 

**Veredito preliminar:** ✅ Viável sem ambiguidade gramatical.

### Q2 — Alternativas de sintaxe

| Opção | Sintaxe | Prós | Contras |
|---|---|---|---|
| **A (proposta PO)** | `from "mod" import Proc1, Proc2` | Familiar (Python-like), declarativa | Requer keyword `import` nova |
| **B — inline decl** | `Proc1 from "mod"` (sem `codes {}`) | Aproveita sintaxe existente de proc | Ambíguo: parece proc normal sem codes; viola a forma atual |
| **C — bloco import** | `import "mod" { Proc1, Proc2 }` | Bloco explícito, consistente | `import` como primeira palavra nunca vista na linguagem |
| **D — use keyword** | `use "mod"` (importa tudo) | Simples | Importação implícita gera conflito de nomes difícil de debugar |

**Recomendação:** Opção A (proposta pelo PO). É a mais legível, não ambígua e consistente com a leitura natural do arquivo. Requer adicionar `KW_IMPORT` ao scanner.

### Q3 — Em qual fase do compilador ocorre a resolução?

**Opções:**

| Fase | Descrição | Risco |
|---|---|---|
| **Parse-time (eager)** | O parser, ao encontrar `from "mod" import ...`, lê e parseia imediatamente `mod.gnj` e injeta os `ProcBlockNode` encontrados | Recursão potencial no parser; dificulta testes unitários |
| **Desugar-time** | Um novo `ImportNode` na AST é resolvido no início de `desugar()` antes da expansão | Separa responsabilidades; desugar já é o lugar de "transformações pré-transpilação" |
| **Fase separada (resolve imports)** | Nova etapa `resolve_imports()` entre parser e desugar; substitui `ImportNode` por `ProcBlockNode` reais | Mais limpo arquiteturalmente; fácil de testar isoladamente |

**Recomendação:** Fase separada `resolve_imports()` — pipeline passa a ser: `Scanner → Parser → ResolveImports → Desugar → Transpiler`. Mantém o parser simples e o desugar agnóstico ao sistema de arquivos.

### Q4 — O que exatamente é importável?

Apenas `ProcBlockNode` (proc-blocos com corpo definido em `.gnj`). Procs normais (`ProcDeclNode`, que são wrappers de macros Jinja2) **não** são importáveis — eles pertencem ao domínio do template, não do código fonte.

Importar um proc normal de outro arquivo não faria sentido semântico: seu `library`/`macro` é relativo ao template, não ao `.gnj`.

### Q5 — Resolução de path

Candidatos para a raiz de resolução:

| Estratégia | Descrição | Risco |
|---|---|---|
| **Relativo ao arquivo compilado** | `from "a.b.c"` resolve para `<dir_do_fonte>/a/b/c.gnj` | Simples; pode levar a paths dispersos |
| **Raiz configurável via CLI** | `--code-base-dir <dir>` | Flexível; necessita mudança na CLI |
| **Relativo ao `code/`** | Mesma convenção do assembler para templates | Consistente com o ecossistema existente; porém code/ é para templates Jinja2, não .gnj |

**Recomendação:** Path relativo ao arquivo compilado como padrão, com opção `--import-base` na CLI para casos avançados.

### Q6 — Problemas a tratar

| Problema | Tratamento sugerido |
|---|---|
| Importar proc com nome já declarado localmente | `ParseError`/`ResolveError` com mensagem clara |
| Arquivo importado não encontrado | `ResolveError` com path resolvido no erro |
| Importação circular (A importa B que importa A) | Detectar ciclo na fase `resolve_imports` via set de arquivos em visita |
| Importar nome não existente no arquivo externo | `ResolveError`: "Proc 'X' não encontrado em 'mod'" |
| Arquivo importado contém procs normais (não-blocos) | Ignorar silenciosamente ou emitir aviso |
| Arquivo importado usa `from "..." import` recursivamente | Resolver recursivamente (suporte a importações em cadeia) |

### Q7 — Impacto no desugar e transpiler

- **Desugar:** após a fase `resolve_imports`, o `ProgramNode` conterá apenas `ProcDeclNode | ProcBlockNode` concretos — idêntico ao estado atual. O desugar **não muda**.
- **Transpiler:** sem mudanças — o transpiler opera sobre a AST pós-desugar que já é canônica.
- **AST IO / ast_io.py:** necessita suporte ao novo nó `ImportNode` para serialização. Após resolução, o `ImportNode` é substituído; ast_io precisa apenas serializar o estado intermediário (pré-resolução) se necessário.

## Esboço do novo nó AST

```python
@dataclass
class ProcImportNode:
    """Nó temporário: importação de proc-blocos de arquivo externo.
    Presente na AST apenas entre parser e resolve_imports.
    Substituído por ProcBlockNode(s) concretos após resolução.
    """
    source_path: str          # caminho dotted: "dir1.dir2.dirn"
    names: list[str]          # nomes dos proc-blocos a importar
```

`ProgramNode.procedures` passa a ser `list[ProcDeclNode | ProcBlockNode | ProcImportNode]` antes da resolução, e `list[ProcDeclNode | ProcBlockNode]` após.

## Risco detalhado

| Risco | Probabilidade | Impacto | Mitigação |
|---|---|---|---|
| Importação circular não detectada | baixa | alto | Set de arquivos visitados + teste de ciclo |
| Conflito de nome silencioso | média | alto | Validação explícita em resolve_imports |
| Ambiguidade gramatical `from` | baixa | alto | Desambiguação pelo contexto (IDENT vs KW_FROM como primeiro token) |
| Path resolution inconsistente entre OS | média | médio | `pathlib.Path` com separador OS-agnóstico |
| Regressão em proc normal existente | baixa | alto | Suite de testes atual cobre casos; verificar manualmente |
