# B-012 — Parser: enforçar ordem `as` antes de `>>` no bloco exec

- Status: pendente
- Prioridade PO: (a definir)
- Owner: agente
- Data de criacao: 2026-05-01
- Sprint: (a definir)

## Contexto

O parser atual aceita as duas ordens abaixo para os modificadores opcionais de um bloco `exec`:

```gnj
exec proc() as "nome" >> variavel { ... }  ← ordem correta (desejada)
exec proc() >> variavel as "nome" { ... }  ← ordem incorreta (deve ser rejeitada)
```

No `compiler/parser.py`, a lógica de parsing no método `_parse_exec_block` trata primeiro
`>> variavel` (linhas ~303–318) e depois `as "nome"` (linhas ~320–324). Isso faz com que a
ordem invertida seja aceita silenciosamente, contrariando a gramática definida em `docs/language.md`.

## Escopo

Corrigir `compiler/parser.py` — método `_parse_exec_block`:

1. Inverter a ordem de parsing: processar `as "nome"` **antes** de `>> variavel`.
2. Garantir que a ordem `>> variavel as "nome"` resulte em `ParseError` com mensagem clara.
3. Atualizar os testes existentes que cubram a sintaxe do `exec` para refletir a ordem correta.
4. Atualizar exemplos (se necessário): `examples/test.gnj` já usa a ordem errada (`>> var as "v"`),
   portanto deve ser corrigido para `as "v" >> var`.

### Gramática esperada após a correção

```
exec_block :=
    'exec' IDENT '(' kwargs ')' ('as' STRING)? ('>>' IDENT)? '{' exec_body '}' while?
```

## Critérios de aceite

- [ ] `exec proc() as "nome" >> var { }` é aceito pelo parser
- [ ] `exec proc() >> var as "nome" { }` lança `ParseError` com mensagem informativa
- [ ] `exec proc() as "nome" { }` (sem `>>`) continua funcionando
- [ ] `exec proc() >> var { }` (sem `as`) continua funcionando
- [ ] `exec proc() { }` (sem nenhum modificador) continua funcionando
- [ ] Todos os testes existentes passam
- [ ] `examples/test.gnj` atualizado para a ordem correta
- [ ] `docs/language.md` revisado para tornar a ordem explícita na gramática formal
