## Objetivo

Implementar suporte à instrução `from "mod" import Name1, Name2` dentro de `procs {}` na linguagem Genjin, permitindo importar proc-blocos definidos em arquivos `.gnj` externos.

---

## Sintaxe aceita

```gnj
procs {
    // procs normais...

    from "Macrosoft.common.ferramentas" import
        Exceção_e_Reinicia,
        Troca_Ferramenta
}
```

O path dotted `"a.b.c"` resolve para `a/b/c.gnj` relativo ao diretório do arquivo compilado (ou ao `--import-base` quando fornecido via CLI).

---

## Decisões de design (B-028)

- **Sintaxe:** Opção A (`from "mod" import Name1, Name2`) — sem ambiguidade gramatical
- **Nova keyword:** `import` → `KW_IMPORT` no scanner
- **Novo nó AST:** `ProcImportNode(source_path, names)` — temporário, pré-resolução
- **Nova fase:** `resolve_imports(ast, base_dir)` entre Parser e Desugar
- **Path base:** diretório do arquivo `.gnj` compilado (padrão); `--import-base` na CLI
- **O que é importável:** apenas `ProcBlockNode` — procs normais não são importáveis
- **Ciclos:** detectados via set de arquivos em visita → `ResolveImportError`

## Escopo detalhado

### S18-T01 — Scanner: keyword `import`

- Adicionar `KW_IMPORT = auto()` em `TokenType`
- Adicionar `'import': TokenType.KW_IMPORT` em `_KEYWORDS`

### S18-T02 — Parser: `ProcImportNode` + parsing

- Novo dataclass `ProcImportNode(source_path: str, names: list[str])`
- Em `_parse_procs`, passo 1: quando token atual é `KW_FROM` (sem IDENT antes) → parsear `from "mod" import Name1, Name2 [,]`
- `ProgramNode.procedures` aceita `ProcImportNode` na pré-resolução
- `_skip_kw_brace_block` / skip interno de `_parse_procs` deve pular import statements
- Placeholder em `declared_procs_map` para nomes importados (sem codes) — validação de codes diferida para desugar

### S18-T03 — `compiler/resolve_imports.py` (arquivo novo)

- Classe `ResolveImportError(Exception)`
- Função pública: `resolve_imports(ast: ProgramNode, base_dir: Path) -> ProgramNode`
  - Substitui cada `ProcImportNode` pelos `ProcBlockNode` concretos do arquivo externo
  - Detecta ciclos (set de paths em visita)
  - Valida nome importado existe no arquivo externo
  - Valida nome não conflita com proc já declarado localmente
  - Resolve importações em cadeia recursivamente

### S18-T04 — Pipeline: integrar `resolve_imports`

- `compiler.py`: inserir `resolve_imports` entre `parse` e `desugar`
- Aceitar `--import-base <dir>` como argumento CLI opcional
- Tratar `ResolveImportError` → stderr + exit 1

### S18-T05 — `ast_io.py`: serialização de `ProcImportNode`

- Importar `ProcImportNode` em `ast_io.py`
- Suporte em `_node_to_dict` e `_node_from_dict`

### S18-T06 — Testes

- `tests/scanner/`: token `import`
- `tests/parser/`: `ProcImportNode` (sintaxe, vírgula final opcional, import vazio = erro)
- `tests/resolve_imports/`: resolução ok, arquivo não encontrado, ciclo, nome não encontrado, conflito de nome, import em cadeia
- Regressão: `pytest` geral sem falhas

### S18-T07 — VS Code extension: highlight de `import`

- Adicionar `import` ao grupo `keyword.declaration.genjin` no `genjin.tmLanguage.json`

### S18-T08 — Atualizar `lenhador-sintetico.gnj`

- Remover a definição inline de `Exceção_e_Reinicia` e `Troca_Ferramenta` do próprio arquivo
- Manter somente o `from "..." import ...` (quando o arquivo externo existir — caso contrário manter como exemplo sintático comentado)
