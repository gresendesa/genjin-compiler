### Mapeamento AST

Todos os pontos de referência a `proc_name` na AST do corpo de um proc-bloco:

| Nó | Campo | Como acessar |
|---|---|---|
| `ExecBlockNode` | `.proc_name` | direto |
| `InlineAtomNode` | `.proc_name` | direto |
| `CaseNode` | `.block` | recursão (ExecBlockNode \| InlineSeqNode) |
| `InlineSeqNode` | `.chained[]` | lista de InlineAtomNode |
| `InlineSeqNode` | `.terminal` | InlineAtomNode \| ExecBlockNode |

Coleta de dependências = DFS simples, ~15 linhas.

### Edge cases analisados

| Caso | Tratamento |
|---|---|
| Dep já local (mesma definição) | Ignorar silenciosamente |
| Dep já local (definição diferente) | `ResolveImportError` — comparação estrutural via `==` de `@dataclass` |
| Auto-referência (`A` usa `A`) | Impossível: desugar do externo já detectou e rejeitou |
| Ciclo `A` usa `B`, `B` usa `A` | Impossível: idem |
| Dep transitiva (proc-bloco que usa proc-bloco) | Injeção recursiva + `visiting_deps: set[str]` como guarda defensiva |
| Dep não encontrada no externo | `ResolveImportError` com nome e arquivo |

### Algoritmo

Duas funções novas em `resolve_imports.py`:
- `_collect_proc_names(body) -> set[str]`: DFS sobre `ExecBlockNode | InlineSeqNode | InlineAtomNode`
- `_inject_deps(pb, external_map, local_names, resolved, visiting_deps)`: injeta deps ausentes, recursivo para `ProcBlockNode`

Modificação em `_resolve_ast`: após importar um nó nomeado, se for `ProcBlockNode` → chamar `_inject_deps`. O `external_map` passa a indexar ambos (`ProcDeclNode | ProcBlockNode`).

**Impacto:** ~50 linhas adicionais em `resolve_imports.py`. Nenhuma outra fase do pipeline é afetada.

## Contexto

B-029 implementou `from "mod" import Name` mas importa apenas o nó nomeado explicitamente, sem suas dependências. Um proc-bloco que chama procs internos (`Teleportar`, `Definir_Home`, etc.) importado isoladamente quebra no desugar/assembler porque esses procs não existem no arquivo importador.

A solução completa é **injeção automática de dependências**: ao importar um proc-bloco, analisar recursivamente seu corpo e puxar todos os procs necessários do arquivo de origem — incluindo procs normais (`ProcDeclNode`) e outros proc-blocos (`ProcBlockNode`) dos quais ele depende.

---

## Objetivo

Ao executar `from "mod" import Troca_Ferramenta`, o compilador deve:

1. Importar `Troca_Ferramenta` (ProcBlockNode)
2. Analisar o corpo de `Troca_Ferramenta` e identificar todos os procs chamados (`Teleportar`, `Definir_Home`, `Guardar_Itens_MKB`, etc.)
3. Para cada dependência encontrada:
   - Se já existe no arquivo importador → não importar (sem conflito)
   - Se existe no arquivo de origem → importar automaticamente
   - Se é outro proc-bloco → analisar suas dependências recursivamente (passo 2)
   - Se não existe em lugar nenhum → erro
4. Garantir que não haja auto-referenciamento nem ciclos entre proc-blocos durante a análise

---

## Sintaxe (sem mudança)

```gnj
procs {
    // Isso:
    from "lenhador" import Troca_Ferramenta

    // ...equivale hoje ao programador escrever manualmente:
    // Teleportar(...) from "..." { codes ... }
    // Definir_Home(...) from "..." { codes ... }
    // Guardar_Itens_MKB(...) from "..." { codes ... }
    // Pegar_Itens_MKB(...) from "..." { codes ... }
    // Troca_Ferramenta(...) { ... }
}
```

## Escopo detalhado

### T01 — Função `_collect_proc_deps` em `resolve_imports.py`

Dado um nó de corpo (`ExecBlockNode | InlineSeqNode`) e o mapa de procs do arquivo externo, retornar o conjunto de nomes de procs referenciados diretamente no corpo (sem recursão — a recursão fica no chamador).

Estrutura a percorrer:
- `ExecBlockNode.proc_name` → nome do proc chamado
- `ExecBlockNode.cases[*].block` → recursão
- `InlineSeqNode.chained` (lista de `InlineAtomNode`) → `atom.proc_name`
- `InlineSeqNode.terminal` → `proc_name` ou `ExecBlockNode`

### T02 — Modificar `_resolve_ast` para injeção automática

Após importar um nó nomeado explicitamente, se for `ProcBlockNode`:
1. Chamar `_collect_proc_deps` no seu corpo
2. Para cada dep não presente em `local_names`:
   - Buscar no mapa de procs do arquivo externo
   - Se `ProcDeclNode` → adicionar diretamente
   - Se `ProcBlockNode` → adicionar e repetir passo 2 recursivamente
3. Usar `visiting_procs: set[str]` (por arquivo externo) para detectar auto-referenciamento e ciclos entre proc-blocos durante a análise de dependências

### T03 — Permitir importar `ProcDeclNode` explicitamente também

Remover o filtro `isinstance(p, ProcBlockNode)` em `external_blocks`. Aceitar ambos os tipos como importáveis explicitamente. (Caso simples de B-030 — a injeção automática faz isso implicitamente, mas o import explícito também deve funcionar.)

### T04 — Testes

- Import de proc-bloco com dependências `ProcDeclNode` → injeção automática
- Import de proc-bloco com dependência em outro proc-bloco → injeção recursiva
- Dep já declarada localmente → não importada (sem conflito)
- Dep não encontrada no arquivo externo → `ResolveImportError`
- Auto-referenciamento no corpo do proc-bloco → `ResolveImportError`
- Import explícito de `ProcDeclNode` → funciona
- Ciclos de proc-blocos via dependências (A usa B que usa A) → `ResolveImportError`

### T05 — `examples/lenhador-sintetico.gnj`

Substituir as declarações manuais redundantes por `from "lenhador" import Troca_Ferramenta, Exceção_e_Reinicia` (ou apenas os proc-blocos), e verificar que as dependências são injetadas automaticamente.

## Decisões de design

- É uma **evolução** de B-029, não rollback
- Injeção automática é **silenciosa** — não gera warnings para procs injetados
- Conflito de nome permanece erro: se o nome importado (explícito ou injetado) já existe localmente com definição diferente, é erro
- A detecção de ciclos de **arquivos** (já existente via `visiting: set[Path]`) é distinta da detecção de ciclos de **dependências de proc-blocos** (nova, via `visiting_procs: set[str]`)
