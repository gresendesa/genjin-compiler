## Contexto

O `code/Federal/@/LenhadorNEO.jinja2` é o programa de produção mais completo do projeto. Ele
está funcionando com o assembler e serve como referência concreta do que o compilador Genjin
deve ser capaz de produzir.

O objetivo deste item é escrever **manualmente** um arquivo `examples/lenhador.gnj` que descreva
o mesmo programa usando a sintaxe `.gnj` — **sem** usar a funcionalidade de proc-blocos (ainda
não implementada). O resultado deve compilar com o pipeline atual e produzir saída equivalente
à estrutura do LenhadorNEO.

Isso servirá como:
1. **Benchmark de qualidade sintática:** o arquivo `.gnj` deve ser legível e fiel ao LenhadorNEO
2. **Caso de teste de referência:** nas sprints B-019/B-020, a expansão de proc-blocos pode ser
   validada comparando a saída com a do lenhador.gnj sem proc-blocos
3. **Prova de que a linguagem atual já é capaz de expressar programas reais**
4. **Referência para o DoD das sprints seguintes** — se o lenhador.gnj compilar sem erros e a
   saída for semanticamente equivalente ao LenhadorNEO, o compilador está correto

                                        }
                                    pass DEMORA_PERCURSO
                                }
                            pass CAMINHO_NÃO_ENCONTRADO, SEM_SUPORTE
                        }
                    case SEM_MAIS_ÁRVORES_PARA_PROCESSAR:
                        exec Reiniciar_Ciclo() {
                            pass REINICIAR_CICLO
                        }
                } while(SUCESSO, CAMINHO_NÃO_ENCONTRADO, DEMORA_EXTRAIR_BLOCO, BLOCO_MUITO_LONGE, ARVORE_NAO_ENCONTRADA, DEMORA_PERCURSO)
            case SEM_ÁRVORES_AO_REDOR:
                exec Notificar_Exceção(mensagem="Não há mais árvores ao redor. Me leve onde haja!") {
                    pass REINICIAR_CICLO
                }
            case FERRAMENTA_GASTA:
                // ← aqui estará o bloco ferramenta_gasta — inline no .gnj (sem proc-bloco)
                // Este é o bloco mais profundo: 7 níveis de exec aninhados
            case SEM_SUPORTE:
                exec Notificar_Exceção(mensagem="Não posso andar ou calcular caminho sobre este bloco. Resolva este problema.") {

                }

## Regra: parâmetros plurais são sempre por referência

**Regra da linguagem (confirmada pelo PO em 2026-05-03):**
Variáveis do tipo plural (`Type[]`) são **sempre** passadas por referência. A notação `&Type[]`
é redundante e não deve ser usada. A sintaxe correta para um parâmetro plural é simplesmente:

```gnj
param: Type[]
```

O compilador deve definir `evaluation='reference'` automaticamente ao detectar `[]` em um
parâmetro. Essa regra está codificada no motor (`code/genjin.jinja2`) e confirmada pela
inspeção do LenhadorNEO (todos os params `CARDINALITY.PLURAL` têm `EVALUATION.REFERENCE`).

**Impacto no parser atual:** `_parse_param_decl` não suporta `[]` — cardinality está fixo em
`'singular'`. Isso precisa ser corrigido como parte desta sprint (S11-T02).
