## Contexto

B-017 especificou a proposta de proc-blocos (DEC-003, `docs/proc-blocos.md`). B-019 cobre a
primeira fase de implementação: a extensão do compilador na camada de parsing.

O parser (`compiler/parser.py`) precisa:
1. Reconhecer proc-blocos na seção `procs { }` (sem `from`, com corpo `{ exec ... }`)
2. Emitir um novo nó AST `ProcBlockNode`
3. Inferir os `output_codes` do proc-bloco a partir do `pass_codes` do bloco raiz interno
4. Ampliar `declared_vars` com os parâmetros `ref` do proc-bloco ao parsear o seu corpo

Este item não toca `desugar.py` — a expansão dos proc-blocos em tempo de compilação fica em
B-020.

## Escopo

### Novo nó AST: `ProcBlockNode`

```python
@dataclass
class ProcBlockNode:
    name: str
    parameters: list[ParamDeclNode]        # placeholders de transpilação
    block: ExecBlockNode | InlineSeqNode   # corpo do proc-bloco
    inferred_codes: list[str]              # inferido dos pass_codes do bloco raiz
```

Posicionamento no arquivo: logo após `ProcDeclNode`, antes de `VarDeclNode`.

### `ProgramNode.procedures` passa a ser `list[ProcDeclNode | ProcBlockNode]`

### `_TYPE_MAP` já tem `TYPE_OBJECT` (B-018 concluído) — nenhuma mudança aqui

### Bifurcação em `_parse_proc_decl`

Renomear `_parse_proc_decl` para continuar retornando `ProcDeclNode`. Criar
`_parse_proc_block` para o novo ramo. Criar função orquestradora `_parse_any_proc_decl`
que decide qual ramo seguir após parsear nome e parâmetros:

```
_parse_any_proc_decl():
    name = IDENT
    ( params )
    → if KW_FROM: _parse_proc_decl_tail(name, params)  # retorna ProcDeclNode
    → if LBRACE:  _parse_proc_block_tail(name, params)  # retorna ProcBlockNode
    → else: ParseError
```

### Dois passos em `_parse_procs`

O corpo de um proc-bloco pode referenciar procs declarados **depois** dele em `procs { }`.
A mesma estratégia de dois passos usada para o `exec` raiz se aplica aqui:

**Passo 1:** percorrer `procs { }` registrando apenas nomes e parâmetros dos proc-blocos
(armazenando posição de token do corpo para parse posterior). Declarar todos os `ProcDeclNode`.

**Passo 2:** com `declared_procs` completo, retornar a cada proc-bloco e parsear seu corpo.

Alternativa mais simples (se a análise mostrar que é segura): parsear o corpo imediatamente, mas
passar para `declared_procs` a lista completa construída no passo 1 (requer lookahead leve). A
decisão final fica a cargo do implementador após análise do código atual de `_parse_procs`.

### `declared_vars` aumentado ao parsear corpo do proc-bloco

Ao parsear o corpo de um `ProcBlockNode`, criar `pseudo_vars` com os nomes dos parâmetros `ref`:

```python
pseudo_vars = declared_vars | {p.name for p in params if p.evaluation == 'reference'}
```

Isso permite que `&home_destino` (parâmetro ref do proc-bloco) passe a validação de variável
de referência sem estar declarado em `vars`.

### Inferência de `inferred_codes`

Após parsear o corpo do proc-bloco, inferir os `output_codes` percorrendo os `pass_codes` do
`ExecBlockNode` raiz:

```python
inferred_codes = list(block.pass_codes)  # pass_codes do ExecBlockNode raiz
```

Armazenar em `ProcBlockNode.inferred_codes`.

### Validação de `>>` proibido no exec raiz do proc-bloco

O `ExecBlockNode` mais externo do corpo de um proc-bloco não pode ter `variable_explicit=True`.
Emitir `ParseError` se `>>` for declarado:

```
ParseError: proc-bloco 'NomeDoBloco' — o exec raiz não pode declarar '>>'; a variável é
herdada do ponto de chamada
```

## Casos de teste (mínimos)

1. **Declaração válida:** proc-bloco com parâmetros `ref` e `lit`, corpo com `exec` simples
2. **Chamada por nome via `@`:** proc-bloco chamado em `case` — o parser resolve nome corretamente
3. **Rejeição `>>`:** exec raiz com `>>` → `ParseError`
4. **Rejeição `Object` ref:** `param: &Object` → `ParseError`
5. **Rejeição cardinalidade plural:** `param: Text[]` em proc-bloco → `ParseError`
6. **Dois passos:** proc-bloco referenciando proc normal declarado após ele em `procs`
7. **Inferência de codes:** `inferred_codes` correto para corpo com múltiplos `pass`

## Risco

Alto. Ponto mais crítico: a estratégia de dois passos em `_parse_procs` (armazenar posição de
token e retomar). O código atual de `_parse_procs` usa posição sequencial; requer análise do
mecanismo de backtracking disponível no parser (se `self._pos` pode ser salvo/restaurado ou se é
necessário um lookahead estruturado).

Mitigação: começar pela abordagem mais simples (pré-coletar nomes em um único passo, parsear
corpos em sequência com `declared_procs` completo), e ajustar se o parser atual não suportar.
