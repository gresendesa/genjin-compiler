## Contexto

Durante a validação do exemplo ilustrativo do B-017 (S10-T02), foi identificado um gap da
linguagem: a sintaxe `itens=["diamond_axe"]` — passagem de uma lista literal como argumento de
`exec` — não está suportada. Os tipos atuais (`Number`, `Text`, `Logic`) só cobrem escalares.

Na prática, procedimentos que interagem com o assembler Jinja2 frequentemente precisam receber
estruturas compostas (listas, dicionários, objetos aninhados) como argumentos literais. Exemplos
reais do Lenhador: `itens: ["diamond_axe"]`, `gaveta: "slot_a"` com metadados extras, etc.

A solução é introduzir o tipo **`Object`** na linguagem DSL — um tipo que representa qualquer
valor estruturado compatível com a notação Jinja2/Python: lista `[...]`, dicionário `{...}`,
ou outro valor composto.

---

## Escopo e restrições

### O que é `Object`

- **Tipo de parâmetro** de procedimento — aparece na declaração de `procs`
- **Só literal** — não existe como `&Object` (referência). Um `Object` nunca aponta para uma
  variável; ele sempre carrega o valor estruturado diretamente na chamada `exec`
- **Não é tipo de variável** — proibido em `vars`. Não faz sentido armazenar um objeto
  estruturado em variável de estado do programa Genjin (que é de runtime do bot/interpretador)
- **Sem cardinalidade plural** (`Object[]` não existe) — o próprio valor já pode ser uma lista

### Semântica de uso

O valor de um argumento `Object` é tratado como **valor Jinja2 opaco**: o transpiler o emite
literalmente na saída, sem transformação. É responsabilidade do programador e do procedimento
receptor interpretar/transformar esse objeto para a linguagem-alvo final (MKB, JS, etc.). O
compilador Genjin não valida a estrutura interna do objeto.

### Exemplos de declaração e uso

```gnj
procs {
    // "itens" é um Object literal — recebe lista, dict, ou qualquer estrutura
    Guardar_Itens(gaveta: Text, itens: Object) from "Federal.@.server_tools.Guardar_Itens" {
        codes SUCESSO<0>, DEMORA<1>
    }

    Configurar(config: Object) from "Federal.@.server_tools.Configurar" {
        codes OK<0>, ERRO<1>
    }
}

exec Guardar_Itens(gaveta="slot_a", itens=["diamond_axe", "iron_axe"]) >> resultado {
    pass SUCESSO, DEMORA
}

exec Configurar(config={"modo": "rapido", "tentativas": 3}) >> resultado {
    pass OK, ERRO
}
```

### O que o transpiler emite

O valor literal do `Object` é emitido diretamente na saída Jinja2, como valor Python/Jinja2
nativo. Exemplos:

| Argumento no `.gnj` | Emitido no Jinja2 |
|---|---|
| `itens=["diamond_axe"]` | `'value': ['diamond_axe']` |
| `config={"modo": "rapido"}` | `'value': {'modo': 'rapido'}` |

> **Requer ação manual do owner (PO/humano):**
>
> O motor `code/genjin.jinja2` precisa suportar um novo tipo `TYPE.OBJECT` (ou equivalente)
> para que os procedimentos receptores possam receber e manipular esses valores. Essa atualização
> é feita manualmente pelo owner e está fora do escopo do compilador.
>
> O compilador pode ser atualizado independentemente, pois ele apenas emite o valor literal sem
> depender do tipo no motor. Porém, o uso em produção só estará completo após a atualização do
> motor.

## Impacto no compilador

### `compiler/scanner.py`
- Adicionar `TYPE_OBJECT = auto()` ao enum `TokenType`
- Adicionar `'Object': TokenType.TYPE_OBJECT` ao dicionário de keywords

### `compiler/parser.py`
- Adicionar `TokenType.TYPE_OBJECT: 'object'` ao `_TYPE_MAP`
- **Bloco `vars`**: rejeitar `TYPE_OBJECT` com `ParseError` (tipo proibido em variáveis)
- **Declaração de parâmetro**: aceitar `TYPE_OBJECT`, mas rejeitar se vier com `&` (referência
  proibida para `Object`) — `ParseError` com mensagem clara
- **Sem cardinalidade plural** (`Object[]`): rejeitar `ParseError`
- `ParamDeclNode` não precisa de modificação (já carrega `type: str`)

### `compiler/transpiler.py`
- Ao emitir argumento de tipo `object`, tratar o `value` do `ArgNode` como valor Jinja2 opaco
  e emiti-lo diretamente (sem aspas adicionais se for lista/dict)
- Adicionar mapeamento `'object'` → `TYPE.OBJECT` no mapa de tipos (se aplicável)

### `compiler/desugar.py`
- Nenhuma mudança esperada — `Object` é apenas um tipo, sem impacto nas transformações de AST

### `vscode-genjin/syntaxes/genjin.tmLanguage.json`
- Adicionar `Object` ao grupo de tipos destacados (sprint separada)

## Sintaxe literal de `Object`

O parser precisará reconhecer listas `[...]` e dicionários `{...}` como valores literais de
argumentos. Isso exige extensão da regra de `arg_value` no parser para cobrir:

```
arg_value ::= NUMBER | STRING | BOOL | list_literal | dict_literal
list_literal ::= '[' (arg_value (',' arg_value)*)? ']'
dict_literal ::= '{' (STRING ':' arg_value (',' STRING ':' arg_value)*)? '}'
```

Esta é a parte de maior risco do item — a gramática de literais aninhados precisa ser cuidadosa
para não criar ambiguidade com a gramática dos blocos `exec {}`.

## Critérios de aceite

- [ ] `Object` aceito como tipo de parâmetro em `procs` (literal apenas)
- [ ] `Object` rejeitado em `vars` com `ParseError` claro
- [ ] `&Object` (referência) rejeitado com `ParseError` claro
- [ ] `Object[]` (plural) rejeitado com `ParseError` claro
- [ ] Lista literal `[...]` e dicionário literal `{...}` reconhecidos como `arg_value`

## Risco

Médio. O risco principal está na gramática de `list_literal` e `dict_literal` no parser: valores
aninhados podem criar ambiguidade com os delimitadores `{` e `}` já usados nos blocos `exec` e
`procs`. Exige análise cuidadosa do lookahead. Mitigado pelo fato de que `{...}` como literal
só ocorre em posição de argumento de chamada, nunca no nível de instrução.
