# B-014 — Parser: tornar a ordem dos blocos `vars`, `procs`, `exec` flexível

- Status: concluído
- Prioridade PO: 1 (crítica)
- Owner: agente
- Data de criacao: 2026-05-01
- Data de conclusao: 2026-05-01
- Sprint: SPR-2026-06

## Contexto

Atualmente o parser exige que os três blocos principais apareçam nesta ordem fixa:

```
program "..." → vars { } → procs { } → exec ... { }
```

A única restrição que deve permanecer é que `program` seja a primeira palavra-chave.
A ordem dos demais blocos (`vars`, `procs`, `exec`) deve ser livre:

```gnj
// Exemplo: exec antes de vars e procs
program "Meu Programa"

exec meu_proc() >> status { ... }

vars {
    status: Number
}

procs {
    meu_proc() from "lib.mod" {
        codes OK<0>
    }
}
```

## Escopo

1. **`compiler/parser.py`** — refatorar `_parse_program` para:
   - Após consumir `program "nome"`, iterar por tokens e despachar `vars`, `procs` e `exec`
     conforme aparecerem (em qualquer ordem).
   - Lançar `ParseError` se um bloco aparecer mais de uma vez.
   - Após coletar os três blocos, realizar a análise semântica (vars declaradas, procs referenciadas).
   - Como `exec` referencia procs e vars declarados nos outros blocos, a validação semântica
     deve ocorrer **depois** que todos os blocos forem coletados (dois passes ou lazy).

2. **`tests/`** — adicionar casos de teste cobrindo ordens alternativas.

3. **`docs/language.md`** — atualizar a seção "Estrutura geral" para indicar que somente
   `program` é posicionalmente obrigatório; os demais blocos podem aparecer em qualquer ordem.

### Restrições que permanecem

- `program` deve ser o primeiro token do arquivo.
- Cada bloco (`vars`, `procs`, `exec`) aparece exatamente uma vez.
- A validação semântica de `exec` (variáveis e procs existem) deve ser feita após todos os blocos serem lidos.

## Critérios de aceite

- [ ] Ordem `program → vars → procs → exec` continua funcionando
- [ ] Ordem `program → procs → vars → exec` é aceita
- [ ] Ordem `program → exec → vars → procs` é aceita
- [ ] Bloco duplicado (ex.: dois `vars`) lança `ParseError`
- [ ] Ausência de qualquer bloco lança `ParseError`
- [ ] Referência a variável/proc não declarada (em qualquer ordem) continua lançando `ParseError`
- [ ] Todos os testes existentes passam
- [ ] `docs/language.md` atualizado

## Dependências

Nenhuma. (Pode ser implementado independentemente de B-012 e B-013.)

## Risco

Médio — a mudança afeta o fluxo principal de `_parse_program` e a validação semântica precisa ser
reestruturada para operar em dois passes (coleta → validação).
