## Descrição

Transformar o compilador Genjin em um pacote Python instalável via `pip`, com pontos de extensão (hooks) para que o consumidor possa substituir os resolvedores padrão por implementações próprias.

Os três resolvedores a expor como extensíveis são:

1. **Resolvedor de templates do compilador** — atualmente busca `genjin.jinja2` / `genjin_fixed.jinja2` via `DottedLoader` no sistema de arquivos. O consumidor deve poder fornecer o conteúdo do template a partir de um banco de dados ou outro repositório.

2. **Resolvedor de templates do assembler** — atualmente lê templates do sistema de arquivos (`code/` tree). O consumidor deve poder fornecer conteúdo de template por nome/caminho a partir de qualquer fonte.

3. **Resolvedor de dependências de importação `.gnj`** — atualmente resolve `from X import Y` lendo arquivos `.gnj` do sistema de arquivos. O consumidor deve poder fornecer o source de um arquivo `.gnj` a partir de um banco de dados ou repositório remoto.

O objetivo final é que outro projeto possa instalar `genjin-compiler` via pip e adaptar esses pontos de extensão à sua realidade sem precisar modificar o código do compilador.

## Critérios de aceite

- [ ] Projeto possui `pyproject.toml` (ou `setup.cfg`) que permite `pip install genjin-compiler` (ou nome equivalente).
- [ ] Existe uma API pública documentada para configurar cada um dos três resolvedores.
- [ ] Os resolvedores possuem implementações padrão que preservam o comportamento atual (filesystem).
- [ ] É possível substituir cada resolvedor de forma independente.
- [ ] Há ao menos um exemplo de uso da API pública com resolvedores customizados.
- [ ] Os testes existentes continuam passando com as implementações padrão.
- [ ] Nenhum contrato de CLI existente é quebrado.

## Escopo técnico

### Empacotamento

- Criar `pyproject.toml` com metadados, dependências e entry points CLI.
- Garantir que `compiler/`, `mkb/` e `assembler.py`/`compiler.py` sejam incluídos no pacote.

### Interface dos resolvedores

Proposta de API (a refinar no planning):

```python
# Resolvedor de template do compilador
class TemplateResolver(Protocol):
    def get_compiler_template(self, name: str) -> str:
        ...

# Resolvedor de template do assembler
class AssemblerTemplateResolver(Protocol):
    def get_template(self, dotted_path: str) -> str:
        ...

# Resolvedor de dependência .gnj
class GnjSourceResolver(Protocol):
    def get_source(self, dotted_path: str) -> str:
        ...
```

Ou pode ser unificado em um único `CompilerConfig` com campos opcionais — decisão a tomar no planning.

### Pontos de injeção

- `compiler/transpiler.py`: injetar `TemplateResolver`.
- `assembler.py` (ou novo `mkb/assembler_core.py`): injetar `AssemblerTemplateResolver`.
- `compiler/resolve_imports.py`: injetar `GnjSourceResolver`.
