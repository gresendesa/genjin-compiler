# B-015 — Compilador: suporte à notação inline `@proc()` (açúcar sintático)

- Status: concluído
- Prioridade PO: 1 (crítica) — atualizado em 2026-05-01
- Owner: agente
- Data de criacao: 2026-05-01
- Sprint: SPR-2026-07 (Fase 1 — análise), SPR-2026-08 (Fase 2 — implementação)

## Contexto

Este item define dois tipos de notação inline para blocos `exec`, funcionando como açúcar
sintático que o compilador deve desaçucarar (desugar) para a forma canônica durante o parsing
ou em uma fase de pré-processamento da AST.

A motivação é reduzir a verbosidade em fluxos aninhados comuns sem perder a expressividade
semântica da linguagem.

---

## Tipo 1 — Execução inline simples (`@proc()`)

### Definição

`@proc(args)` em posição de body de `case` (ou no nível raiz do `exec`) é expandido para um
bloco `exec` que passa todos os códigos de saída do procedimento referenciado.

### Exemplos

**Exemplo 1.1 — nível raiz:**

```gnj
// Notação inline
@inicia_processo() >> status_var while(ERROR)

// Equivalente canônico
exec inicia_processo() >> status_var {
    pass OK
} while(ERROR)
```

> Nota: `pass OK` porque `OK` é o único código não coberto por `while(ERROR)`.

**Exemplo 1.2 — dentro de `case`:**

```gnj
// Notação inline
exec inicia_processo() >> status_var {
    case OK: @inicia_processo()
    pass ERROR, OK
} while(ERROR)

// Equivalente canônico
exec inicia_processo() >> status_var {
    case OK: exec inicia_processo() {
        pass ERROR, OK
    }
    pass ERROR, OK
} while(ERROR)
```

> Nota: o bloco gerado pelo `@` herda os códigos que o pai precisa ver borbulhados (pass do pai).

---

## Tipo 2 — Execução inline com case unário encadeado (`@proc() when(CODE)`)

### Definição

`@proc() when(CODE)` expande para um bloco `exec` que contém um `case CODE:` apontando para o
próximo `@` ou `exec` na sequência. Os demais códigos são passados.

A sequência de anotações `@` dentro de um mesmo `case` é tratada como **encadeamento linear**:
cada `@` com `when(CODE)` gera um `case CODE:` aninhado dentro do anterior.

**Ordem obrigatória dentro de um átomo:** `[>>]` → `[while(...)]` → `[when(...)]`.

O `when(CODE)` vem após o `while` pois é a **condição de saída** do bloco rumo ao próximo:
o bloco executa, repete em `while(W)` quando necessário, e somente quando `when(CODE)` ocorre
prossegue para o bloco seguinte da sequência.

### Exemplos

**Exemplo 2.1 — encadeamento com `while` e inline simples:**

```gnj
// Notação inline
@inicia_processo() >> status_var while(ERROR) when(OK)
exec inicia_processo() {
    case OK: exec inicia_processo() {
        pass ERROR, OK
    }
    pass OK
} while(ERROR)

// Equivalente canônico
exec inicia_processo() >> status_var {
    case OK: exec inicia_processo() {
        case OK: exec inicia_processo() {
            pass ERROR, OK
        }
        pass OK
    } while(ERROR)
    pass OK
} while(ERROR)
```

**Exemplo 2.2 — múltiplas anotações dentro de `case`:**

```gnj
// Notação inline
exec inicia_processo() >> status_var {
    case OK: @inicia_processo() when(OK)
             @inicia_processo() while(ERROR) when(ERROR)
             @inicia_processo()
    pass ERROR, OK
} while(ERROR)

// Equivalente canônico
exec inicia_processo() >> status_var {
    case OK: exec inicia_processo() {
        case OK: exec inicia_processo() {
            case ERROR: exec inicia_processo() {
                pass ERROR, OK
            }
        } while(ERROR)
        pass OK
    }
    pass ERROR, OK
} while(ERROR)
```

### Fase obrigatória: análise de viabilidade e design

Antes de qualquer implementação, produzir um documento de análise técnica cobrindo:

1. **Gramática formal** — expressar os dois tipos de `@` na gramática EBNF da linguagem.
2. **Regras de desaçucaramento** — definir formalmente como cada padrão é expandido para AST canônica:
   - Como inferir os `pass` gerados automaticamente (quais códigos são passados).
   - Como resolver herança de variável (`>>`) nas expansões.
   - Como lidar com `while` nas notações `@`.
3. **Análise de intuitividade** — avaliar a legibilidade para humanos:
   - A notação `@proc()::CODE` é intuitiva? Conflita com outros usos de `@` em linguagens populares?
   - A sequência de `@` no mesmo `case` representa bem o encadeamento?
4. **Comparação com mercado** — avaliar como padrões similares são tratados em:
   - Linguagens de fluxo (ex.: Apache Camel DSL, Temporal Workflows, AWS Step Functions ASL).
   - Linguagens com açúcar sintático para monads/chains (Haskell `do`, Rust `?`, Kotlin `let`).
5. **Riscos de ambiguidade** — identificar casos onde a expansão pode gerar AST inesperada.
6. **Proposta de gramática alternativa** — se houver problema de intuitividade, propor variante.

### S7-T01 — Gramática EBNF formal

A gramática atual (fragmento relevante):

```ebnf
(* posição raiz do programa *)
root_exec     ::= 'exec' proc_call [alias] [arrow] '{' exec_body '}' ['while' '(' IDENT ')']

(* corpo do exec *)
exec_body     ::= (case_stmt | pass_stmt)+
case_stmt     ::= 'case' IDENT ':' root_exec
pass_stmt     ::= 'pass' IDENT (',' IDENT)*

(* auxiliares *)
proc_call     ::= IDENT '(' [arg_list] ')'
alias         ::= 'as' STRING
arrow         ::= '>>' IDENT
```

Adições necessárias para suportar a notação `@`:

```ebnf
(* === Notação inline === *)

(* Átomo inline: base de ambos os tipos
   Ordem obrigatória: >> antes de while; while antes de when *)
inline_atom   ::= '@' IDENT '(' [arg_list] ')' [arrow] ['while' '(' IDENT ')'] ['when' '(' IDENT ')']

(* Sequência inline: 1 ou mais átomos; restrições semânticas abaixo *)
inline_seq    ::= inline_atom+

(* === Posições onde inline é aceito === *)

(* Substitui root_exec no nível do programa *)
root_exec     ::= exec_canonical
               | inline_seq

(* Corpo de um case pode ser exec canônico OU sequência inline *)
case_stmt     ::= 'case' IDENT ':' (exec_canonical | inline_seq)

(* exec canônico é o atual root_exec renomeado *)
exec_canonical ::= 'exec' proc_call [alias] [arrow] '{' exec_body '}' ['while' '(' IDENT ')']
```

**Restrições semânticas (não capturáveis em EBNF pura):**

| # | Restrição |
|---|---|
| R1 | Numa `inline_seq`, somente o **último** átomo deve omitir `when(CODE)` (ser terminal). |
| R2 | Numa `inline_seq`, todos os átomos não-últimos **devem** ter `when(CODE)`. |
| R3 | Uma `inline_seq` de um único átomo deve ser terminal (sem `when`) — equivale ao Tipo 1. |
| R4 | `>>` em qualquer átomo aplica-se apenas ao bloco gerado **por aquele átomo**. |
| R5 | `while(W)` em qualquer átomo aplica-se ao bloco gerado por aquele átomo, **antes** de `when`. |
| R6 | O `when(CODE)` de cada átomo não-último deve nomear um código válido declarado no `proc` do átomo. |
| R7 | Ordem obrigatória dentro de um átomo: `[>>]` → `[while(...)]` → `[when(...)]`. |

**Exemplos com a sintaxe aprovada:**

```gnj
// Tipo 1 — inline simples (sem when)
@inicia_processo() >> status_var while(ERROR)

// Tipo 2 — encadeamento com when(CODE)

// Exemplo 2.1
@inicia_processo() >> status_var while(ERROR) when(OK)
exec inicia_processo() {
    case OK: exec inicia_processo() {
        pass ERROR, OK
    }
    pass OK
} while(ERROR)

// Exemplo 2.2 — múltiplos átomos em case
case OK: @inicia_processo() when(OK)
         @inicia_processo() while(ERROR) when(ERROR)
         @inicia_processo()
```

### S7-T02 — Regras formais de desaçucaramento

#### Definições

- `pass_context`: conjunto de códigos que o contexto pai exige que sejam borbulhados (`pass_codes` do `exec` pai).
- `inherited_var`: variável herdada do bloco `exec` pai (pode ser `None` na raiz).
- `codes(proc)`: conjunto de códigos declarados em `codes` da declaração do `proc`.

#### Algoritmo: `desugar_sequence(atoms, pass_context, inherited_var) → ExecBlockNode`

O desugar percorre a sequência da **direita para a esquerda**, construindo os nós de dentro para fora.

```
função desugar_sequence(atoms, pass_context, inherited_var):

  1. inner ← desugar_terminal(atoms[-1], pass_context, inherited_var)

  2. para cada atom em reversed(atoms[:-1]):
       garantir que atom.chain_code existe (R2)
       inner ← desugar_chained(atom, inner, inherited_var)

  3. retornar inner
```

#### `desugar_terminal(atom, pass_context, inherited_var) → ExecBlockNode`

Expande o átomo **final** (simples, sem `::CODE`):

```
var       = atom.var se declarado, senão inherited_var
while_set = { atom.while_code } se while declarado, senão ∅
pass_set  = pass_context − while_set

retornar ExecBlockNode(
  proc_name        = atom.proc,
  kwargs           = atom.args,
  variable         = var,
  variable_explicit= (atom.var é declarado),
  block_name       = None,
  cases            = [],
  loop_while       = lista(while_set),
  pass_codes       = lista(pass_set),
)
```

#### `desugar_chained(atom, inner, inherited_var) → ExecBlockNode`

Expande um átomo com `when(CODE)`, encapsulando `inner` como `case`:

```
var       = atom.var se declarado, senão inherited_var
while_set = { atom.while_code } se while declarado, senão ∅
when_code = atom.when_code       (* código de encadeamento *)
pass_set  = codes(atom.proc) − { when_code } − while_set

retornar ExecBlockNode(
  proc_name        = atom.proc,
  kwargs           = atom.args,
  variable         = var,
  variable_explicit= (atom.var é declarado),
  block_name       = None,
  cases            = [ CaseNode(output_code=when_code, block=inner) ],
  loop_while       = lista(while_set),
  pass_codes       = lista(pass_set),
)
```

#### Observações críticas

| Observação | Consequência |
|---|---|
| `pass_set` do átomo chained depende de `codes(proc)` | O desugar precisa de acesso à tabela de procs — confirmando que é uma fase **pós-parser**. |
| `pass_context` do terminal é herdado do pai | O desugar precisa receber `pass_context` ao entrar num `case`, não pode ser calculado em isolamento. |
| `>>` não precisa aparecer em todos os átomos | Apenas o primeiro átomo da chain costuma ter `>>`. Os demais herdam. |
| `while` pode aparecer em qualquer átomo da chain | Cada átomo gera seu próprio `ExecBlockNode` com `loop_while` independente. |

### S7-T03 — Análise de intuitividade

#### `@proc()` — Tipo 1

**Favorável:** o `@` como prefixo de "invocação especial/decorada" já existe em Python (decorators) e Java (anotações). A leitura `@proc()` como "execute este proc de forma inline" é razoável.

**Neutro:** não conflita com usos comuns de `@` em linguagens orientadas a objeto (Python usa `@` para decorators de função/classe, não para chamadas inline em fluxos).

**Veredicto:** intuitivo para o Tipo 1. Risco baixo.

#### `@proc()::CODE` — Tipo 2

**Desfavorável:**
- `::` é o operador de escopo em C++/Rust (`std::string`, `Vec<T>`), o operador de namespace em Ruby (`Module::Class`) e o operador de tipo em Haskell/Rust.
- Nenhum desses usos sugere "encadear condicional ao código `CODE`".
- A leitura natural de `@proc()::CODE` seria "acessar o membro `CODE` de `proc()`" (scope/member access), não "continuar no código CODE".

**Alternativa mais legível:** `@proc() when(CODE)` — alinha com `while(CODE)` já presente na linguagem. Ambos são condições baseadas em código de saída, apenas com semântica diferente (`while` = repetir, `when` = encadear).

**Veredicto:** `::CODE` tem risco médio/alto de confusão. Recomendo avaliar `when(CODE)` como alternativa.

### S7-T04 — Comparação com mercado

| Referência | Padrão de encadeamento condicional | Equivalência |
|---|---|---|
| **Apache Camel DSL** | `.choice().when(predicate).to(uri).otherwise().to(uri)` | `when` explícito; mais próximo de `when(CODE)` |
| **AWS Step Functions ASL** | `"Next": "StateName"` + `"Catch": [{"ErrorEquals": [...], "Next": "..."}]` | Sem açúcar; sempre forma canônica |
| **Temporal Workflows** | Código normal da linguagem host; sem DSL inline | Não aplicável |
| **Haskell `do` notation** | `x <- action; case x of { OK -> next; _ -> pass }` | `>>=` com pattern match; sem símbolo dedicado |
| **Rust `?` operator** | `let x = action()?;` propaga `Err`, continua em `Ok` | Encadeia no "caminho feliz" (OK); sem código explícito |
| **Kotlin `let`/`also`** | `result.let { if (it == OK) next(it) }` | Lambda; sem símbolo especial |

**Conclusão:** nenhuma referência de mercado usa `::` para encadeamento condicional por código. O padrão mais próximo é o `when(predicate)` do Camel e o `?` do Rust (que fixa o código como `Ok`). A notação `when(CODE)` é consistente com esses padrões e com o idioma já estabelecido na linguagem (`while(CODE)`).

### S7-T05 — Riscos de ambiguidade

| # | Cenário | Risco | Detalhamento |
|---|---|---|---|
| A1 | `@proc()::CODE >> var` — `>>` aplica-se ao átomo ou à chain inteira? | Médio | Pela regra R4, aplica-se apenas ao átomo. Deve ser documentado claramente; o desugar deve respeitar isso. |
| A2 | `@proc() while(W)` como átomo **não-terminal** | Alto | Um átomo não-terminal com `while` gera um `ExecBlockNode` com `loop_while`. O átomo seguinte (via `::CODE` do átomo ANTERIOR) é aninhado dentro. Isso é semanticamente correto, mas visualmente pode parecer que `while` "quebra" o encadeamento. |
| A3 | Terminal `@proc()` com `pass_context` vazio | Médio | Se o pai não tem `pass`, o terminal gera `pass_codes=[]`. Isso pode ser inválido semanticamente (exec sem pass nem while pode não cobrir todos os códigos do proc). Exige validação estática. |
| A4 | `inline_seq` de um único átomo com `::CODE` e sem sucessor | Alto | Viola R2. Deve ser erro de parse com mensagem clara: "átomo encadeado sem sucessor". |
| A5 | Herança de `pass_context` em cadeia profundamente aninhada | Médio | O `pass_context` do terminal deve ser o `pass_codes` do `exec` pai IMEDIATO que contém o `case` onde a cadeia está. Em chains profundas isso deve ser passado explicitamente pelo desugarer. |
| A6 | `@proc()` no nível raiz com `pass_context` implícito | Baixo | No nível raiz não há pai; o `pass_context` raiz é vazio (sem pai para borbulhar). O Tipo 1 na raiz gera `pass_codes = ∅`. Isso parece correto: o exec raiz não borbulha para ninguém. |

### S7-T06 — Proposta de gramática alternativa

**Decisão tomada pelo PO:** adotar `when(CODE)` com ordem `[while(W)] when(CODE)`.

```gnj
// Átomo com encadeamento
@proc(args) [>> var] [while(W)] when(CODE)

// Exemplos
@inicia_processo() >> status_var while(ERROR) when(OK)
@inicia_processo() when(OK)
@inicia_processo() while(ERROR) when(ERROR)
@inicia_processo()   // terminal — sem when
```

**Semântica de `while → when`:** o bloco executa, repete em `while(W)`, e **quando** o código `when(CODE)` ocorre, prossegue para o bloco seguinte da sequência. Os demais códigos (nem `while` nem `when`) são passados para o pai.

**Token novo necessário:** `KW_WHEN` no scanner.

#### Comparação final das opções

| Critério | `::CODE` (descartada) | `when(CODE)` (aprovada) |
|---|---|---|
| Consistência com `while(CODE)` | Baixa | **Alta** |
| Risco de confusão (`::` = scope em C++/Rust) | Médio/alto | **Baixo** |
| Verbosidade | Menor | Maior |
| Ordem natural na leitura | "procede no CODE" | "executa [loop em W] quando CODE vai pro próximo" |
| Mudança no scanner | `DOUBLE_COLON` | `KW_WHEN` |
