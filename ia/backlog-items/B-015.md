## Contexto

Este item define dois tipos de notação inline para blocos `exec`, funcionando como açúcar
sintático que o compilador deve desaçucarar (desugar) para a forma canônica durante o parsing
ou em uma fase de pré-processamento da AST.

A motivação é reduzir a verbosidade em fluxos aninhados comuns sem perder a expressividade
semântica da linguagem.

---

## Tipo 1 — Execução inline simples (`@proc()`)

### Definição

`@proc(args)` em posição de body de `case` (ou no nível raiz do `exec`) é expandido para um
bloco `exec` que passa todos os códigos de saída do procedimento referenciado.

### Exemplos

**Exemplo 1.1 — nível raiz:**

```gnj
// Notação inline
@inicia_processo() >> status_var while(ERROR)

// Equivalente canônico
exec inicia_processo() >> status_var {
    pass OK
} while(ERROR)
```

> Nota: `pass OK` porque `OK` é o único código não coberto por `while(ERROR)`.

**Exemplo 1.2 — dentro de `case`:**

```gnj
// Notação inline
exec inicia_processo() >> status_var {
    case OK: @inicia_processo()
    pass ERROR, OK
} while(ERROR)

// Equivalente canônico
exec inicia_processo() >> status_var {
    case OK: exec inicia_processo() {
        pass ERROR, OK
    }
    pass ERROR, OK
} while(ERROR)
```

> Nota: o bloco gerado pelo `@` herda os códigos que o pai precisa ver borbulhados (pass do pai).

## Tipo 2 — Execução inline com case unário encadeado (`@proc()::CODE`)

### Definição

`@proc()::CODE` expande para um bloco `exec` que contém um `case CODE:` apontando para o
próximo `@` ou `exec` na sequência. Os demais códigos são passados.

A sequência de anotações `@` dentro de um mesmo `case` é tratada como **encadeamento linear**:
cada `@` com `::CODE` gera um `case CODE:` aninhado dentro do anterior.

### Exemplos

**Exemplo 2.1 — encadeamento com `while` e inline simples:**

```gnj
// Notação inline
@inicia_processo()::OK >> status_var while(ERROR)
exec inicia_processo() {
    case OK: exec inicia_processo() {
        pass ERROR, OK
    }
    pass OK
} while(ERROR)

// Equivalente canônico
exec inicia_processo() >> status_var {
    case OK: exec inicia_processo() {
        case OK: exec inicia_processo() {
            pass ERROR, OK
        }
        pass OK
    } while(ERROR)
    pass OK
} while(ERROR)
```

**Exemplo 2.2 — múltiplas anotações dentro de `case`:**

```gnj
// Notação inline
exec inicia_processo() >> status_var {
    case OK: @inicia_processo()::OK
             @inicia_processo()::ERROR while(ERROR)
             @inicia_processo()
    pass ERROR, OK
} while(ERROR)

// Equivalente canônico
exec inicia_processo() >> status_var {
    case OK: exec inicia_processo() {
        case OK: exec inicia_processo() {
            case ERROR: exec inicia_processo() {
                pass ERROR, OK
            }
        } while(ERROR)
        pass OK
    }
    pass ERROR, OK
} while(ERROR)
```

## Escopo deste item

### Fase obrigatória: análise de viabilidade e design

Antes de qualquer implementação, produzir um documento de análise técnica cobrindo:

1. **Gramática formal** — expressar os dois tipos de `@` na gramática EBNF da linguagem.
2. **Regras de desaçucaramento** — definir formalmente como cada padrão é expandido para AST canônica:
   - Como inferir os `pass` gerados automaticamente (quais códigos são passados).
   - Como resolver herança de variável (`>>`) nas expansões.
   - Como lidar com `while` nas notações `@`.
3. **Análise de intuitividade** — avaliar a legibilidade para humanos:
   - A notação `@proc()::CODE` é intuitiva? Conflita com outros usos de `@` em linguagens populares?
   - A sequência de `@` no mesmo `case` representa bem o encadeamento?
4. **Comparação com mercado** — avaliar como padrões similares são tratados em:
   - Linguagens de fluxo (ex.: Apache Camel DSL, Temporal Workflows, AWS Step Functions ASL).
   - Linguagens com açúcar sintático para monads/chains (Haskell `do`, Rust `?`, Kotlin `let`).
5. **Riscos de ambiguidade** — identificar casos onde a expansão pode gerar AST inesperada.
6. **Proposta de gramática alternativa** — se houver problema de intuitividade, propor variante.

### Fase de implementação (após aprovação da análise)

1. **`compiler/scanner.py`** — adicionar token `AT` (`@`) se ainda não existir.
2. **`compiler/parser.py`** — implementar parsing das notações `@` nos contextos válidos.
3. **`compiler/transpiler.py`** — verificar se precisa de ajuste (deve ser transparente se a AST for idêntica à canônica).
4. **`tests/`** — adicionar testes para cada padrão do Tipo 1 e Tipo 2.
5. **`docs/language.md`** — documentar a notação inline como seção dedicada.
6. **`examples/`** — adicionar exemplo demonstrando os dois tipos.
