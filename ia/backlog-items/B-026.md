## Motivação

`code/genjin.jinja2` é o motor central do Genjin — INTOCÁVEL e sem cobertura de testes
automatizados. Qualquer bug silencioso (como o EXP-005) só é descoberto em exemplos reais.
Este item cria uma suite completa que torna cada regra da especificação `docs/genjin.md`
verificável de forma reproduzível.

---

## Escopo

### Testes Unitários (por macro)

Cada teste é um template `.jinja2` mínimo que importa `genjin` (ou `genjin_fixed`),
monta os dados necessários e invoca diretamente a macro sob teste. A saída é verificada
por texto ou pelo não-lançamento de exceção.

| # | Macro | Casos de teste |
|---|---|---|
| U-01 | `filter_list_objects` | lista vazia; item encontrado; item não encontrado; `attribute_of_attribute` |
| U-02 | `is_list` | lista; string; dicionário; número |
| U-03 | `is_dict` | dicionário; lista; string |
| U-04 | `macromod_variable_renderer` | TYPE.TEXT (`&` prefix); TYPE.NUMBER (`#` prefix); TYPE.LOGIC (sem prefix); tipo inválido (deve levantar exceção) |
| U-05 | `macromod_set_variable` | TYPE.TEXT (aspas duplas); TYPE.NUMBER (sem aspas); TYPE.LOGIC |
| U-06 | `macromod_get_variable_value` | variável NUMBER; variável TEXT |
| U-07 | `macromod_decision_structure_variable_value_equality_renderer` | TYPE.NUMBER (`==`); TYPE.TEXT (`=="..."`) |
| U-08 | `macromod_decision_structure_variable_value_disjunction_renderer` | 1 valor; N valores; lista vazia (sem output) |
| U-09 | `macromod_render_conditional_loop` | `codes` não vazio (DO/WHILE); `codes` vazio (sem loop) |
| U-10 | `get_unhandled_codes` | sem cases, sem while → todos; com case cobrindo 1; com while cobrindo 1 |
| U-11 | `get_unhandled_codes_recursively` | **caso crítico EXP-005**: filho passa código absorvido por LOOP_WHILE do pai; múltiplas variáveis; sem cases |
| U-12 | `checks_code_handling` | Regra #1 OK; Regra #1 falha (código não tratado, não passado); Regra #2 OK com bloco superior; Regra #2 falha sem bloco superior |
| U-13 | `CALL_PROCEDURE` | literal→literal; reference→literal (get_value); reference→reference; OBJECT; plural; sem parâmetros; parâmetro default; erros esperados (tipo errado, plural literal) |
| U-14 | `RENDER_BLOCK` | bloco sem cases; bloco com 1 case; bloco com LOOP_WHILE; herança de variável (filho sem VARIABLE) |

### Testes de Integração

Programas completos em notação Jinja2 (objeto `program` + `build()`), executados via
`python assembler.py <template> -d code/`. A saída gerada é comparada com golden files.

| # | Cenário | Regra validada |
|---|---|---|
| I-01 | Proc sem parâmetros, 1 código, sem cases | pipeline mínimo |
| I-02 | Proc com parâmetro literal NUMBER | CALL_PROCEDURE literal |
| I-03 | Proc com parâmetro por referência TEXT | CALL_PROCEDURE reference |
| I-04 | Proc com parâmetro OBJECT | TYPE.OBJECT |
| I-05 | Bloco com LOOP_WHILE | DO/WHILE gerado corretamente |
| I-06 | Bloco com 2 CASES | IF aninhado |
| I-07 | Hierarquia 3 níveis, herança de variável | herança implícita de VARIABLE |
| I-08 | Código borbulhado absorvido por LOOP_WHILE intermediário (**caso EXP-005**) | Regra #2 + fix SPR-2026-15 |
| I-09 | PASS_CODES válido → não lança exceção | Regra #1 positivo |
| I-10 | Código não tratado sem PASS_CODES → exceção esperada | Regra #1 negativo |
| I-11 | Código borbulhado sem LOOP_WHILE e sem PASS_CODES no raiz → exceção | Regra #2 negativo |
| I-12 | Programa equivalente ao `gui.mkb` (smoke test do exemplo canônico) | regressão geral |

## Abordagem técnica

- Os templates de teste importam `"genjin"` normalmente (resolvido pelo DottedLoader em `code/`).
- `conftest.py` expõe fixture `run_genjin(template, expect_error=False)`:
  - executa `assembler.py <template> -d code/` via `subprocess`
  - retorna stdout se `expect_error=False`
  - captura e retorna a mensagem de exceção se `expect_error=True`
- Comparação de output: normalização de whitespace antes de comparar com golden files.
- Testes negativos (exceção esperada) verificam substring da mensagem de erro.
- Nenhum teste modifica `code/genjin.jinja2`.

## Risco

- **Médio-alto:** o maior risco é a criação dos golden files — qualquer saída incorreta aceita como golden congela um bug silencioso.
  - Mitigação: revisão manual de cada golden file pelo PO antes do aceite.
- **Médio:** templates de teste unitário exigem montar CORE mock (VARIABLE_MANAGER, PROCEDURE_MANAGER, etc.) em Jinja2 puro — verboso mas viável usando dicionários inline.
