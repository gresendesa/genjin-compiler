# B-027 — Parser: literais de coleção (`[...]` / `{...}`) como argumentos de parâmetros `Object`

- Status: concluído
- Prioridade PO: 1 (crítica)
- Owner: gresendesa
- Risco: médio
- Data de criação: 2026-05-04
- Sprint: SPR-2026-17

---

## Contexto

Atualmente, parâmetros do tipo `Object` só aceitam como argumento uma `STRING` (valor entre
aspas duplas). O programador precisa "escapar" a coleção dentro de uma string:

```gnj
Guardar_Itens_MKB(gaveta="gaveta_mkb_guardar", itens="['diamond_axe']", ...)
```

Isso é inconveniente e propenso a erro, especialmente em coleções maiores ou dicionários aninhados.
A intenção é permitir a sintaxe literal nativa, sem aspas e com formatação livre (multilinhas):

```gnj
Guardar_Itens_MKB(
    gaveta="gaveta_mkb_guardar",
    itens=['diamond_axe'],
    ...
)

Configurar(config={
    'modo': 'rapido',
    'tentativas': 3
})
```

O transpiler já emite o valor de `Object` como valor "raw" (sem aspas no Jinja2 resultante);
o único obstáculo é o scanner/parser não reconhecerem a sintaxe literal de coleção.

---

## Escopo

### O que muda

| Componente | Mudança |
|---|---|
| `compiler/scanner.py` | Novo `TokenType.OBJECT_LITERAL`. Ao encontrar `[` ou `{` dentro de `(...)` de argumentos, captura o texto bruto até o delimitador balanceado correspondente (suportando aninhamento e quebras de linha). |
| `compiler/parser.py` | Em `_parse_kwargs`, aceitar `OBJECT_LITERAL` (além de `STRING`) quando `param.type == 'object'`. O valor capturado é emitido como `raw=True`. |
| `compiler/parser.py` | Métodos `_skip_exec_block` / `_skip_inline_seq` devem tratar o novo token naturalmente (já avançam token a token, então nenhuma mudança é necessária se o scanner emitir um único token). |
| `examples/lenhador.gnj` | Migrar argumentos `Object` de string escapada para literal nativo. |

### O que NÃO muda

- Semântica do transpiler (`raw=True` já funciona).
- `genjin.jinja2` (motor) — já trata `TYPE.OBJECT` corretamente.
- Parâmetros de outros tipos (`Text`, `Number`, `Logic`, `&Tipo`) — inalterados.
- `Object` continua proibido em `vars` e como `&Object` (referência).

---

## Detalhes de implementação

### Scanner — `OBJECT_LITERAL`

Novo estado no método `_next_token`: ao encontrar `[` ou `{` (fora de string/comentário),
captura caracteres até o fechamento balanceado, preservando quebras de linha e espaços.
Exemplo de captura:

```
['diamond_axe', 'iron_axe']
```
→ `Token(OBJECT_LITERAL, "['diamond_axe', 'iron_axe']", line=N)`

```
{
    'modo': 'rapido',
    'tentativas': 3
}
```
→ `Token(OBJECT_LITERAL, "{\n    'modo': 'rapido',\n    'tentativas': 3\n}", line=N)`

**Casos de borda:**
- Aninhamento: `[['a', 'b'], ['c']]` — rastrear profundidade.
- Aspas internas: `{'chave': 'valor com [colchetes]'}` — dentro de aspas, não conta como delimitador.
- Quebras de linha: atualizar `self._line` normalmente para rastreamento de erros.

### Parser — `_parse_kwargs`

Substituir o trecho atual:

```python
if val_tok.type == TokenType.STRING:
    self._advance()
    raw = param.type == 'object'
    kwargs[...] = ArgNode(value=val_tok.value, evaluation='literal', raw=raw)
```

Por:

```python
if val_tok.type == TokenType.STRING:
    self._advance()
    raw = param.type == 'object'
    kwargs[...] = ArgNode(value=val_tok.value, evaluation='literal', raw=raw)
elif val_tok.type == TokenType.OBJECT_LITERAL:
    if param.type != 'object':
        raise ParseError(
            f"argumento '{name_tok.value}': literal de coleção só é permitido para parâmetros Object",
            val_tok.line,
        )
    self._advance()
    kwargs[...] = ArgNode(value=val_tok.value, evaluation='literal', raw=True)
```

## Critérios de aceite

- [ ] `itens=['diamond_axe']` e `itens={"chave": "valor"}` compilam sem erro quando o parâmetro for `Object`
- [ ] Formatação multilinhas é aceita (newlines e indentação preservadas no token)
- [ ] Para parâmetros não-`Object`, usar `[...]` ou `{...}` como argumento levanta `ParseError` descritivo
- [ ] Strings com colchetes/chaves internas continuam funcionando: `itens="['diamond_axe']"` → `raw=True`
- [ ] `examples/lenhador.gnj` migrado para sintaxe literal e compilando sem erro
- [ ] Testes unitários (scanner + parser) e testes de integração E2E existentes passam (0 regressões)
