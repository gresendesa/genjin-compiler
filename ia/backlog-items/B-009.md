# B-009 — Permitir que o assembler leia o template da entrada padrão (stdin)

- Status: em andamento (SPR-2026-04)
- Prioridade PO: 2 (alta)
- Owner: agente
- Data de criacao: 2026-04-28
- Ultima atualizacao: 2026-04-28

## Contexto

O assembler hoje exige um arquivo de template em disco (`python assembler.py template.jinja2 -d dir -o out`).

O compilador (`compiler.py`) gera o template Jinja2 para stdout, o que torna natural o pipe:

```bash
python compiler.py programa.gnj | python assembler.py - -d ./code -o saida.txt
```

Sem suporte a stdin no assembler, o usuário precisa salvar o template em arquivo temporário antes de renderizá-lo, quebrando o fluxo de pipe.

## Escopo

Permitir que o argumento `template` aceite `-` como valor especial indicando leitura de stdin.

Quando `template == '-'`:
1. Ler o conteúdo do template de `sys.stdin`.
2. Renderizar usando `Environment` com `BaseLoader` (ou `DictLoader`) em vez de `FileSystemLoader`, pois não há arquivo em disco.
3. `extends`, `include` e `import` dentro do template lido pelo stdin devem continuar sendo resolvidos pelo `--templates-dir` informado.
4. O argumento `-d/--templates-dir` continua obrigatório quando o template referencia outros arquivos.

## Comportamento esperado

```bash
# Pipeline completo: .gnj → template → saída renderizada
python compiler.py programa.gnj | python assembler.py - -d ./code -o saida.txt

# Template inline sem extends/include (templates-dir pode ser omitido)
echo '{{ nome }}' | python assembler.py - -v nome=Mundo -o saida.txt
```

## Critérios de aceite

- [ ] `python assembler.py - -d ./code -o out.txt` lê o template de stdin e renderiza corretamente.
- [ ] `extends`, `import` e `include` dentro do template stdin resolvem arquivos relativos a `--templates-dir`.
- [ ] Quando `-d` é omitido e o template stdin não usa `extends`/`include`/`import`, a renderização funciona.
- [ ] Quando `-d` é omitido e o template stdin usa `extends`/`include`/`import`, emite erro claro para stderr (exit 2).
- [ ] Pipeline `compiler.py | assembler.py -` funciona end-to-end com `basic.gnj`.
- [ ] Testes automatizados cobrem: leitura de stdin, resolução de extends, erro sem -d.
- [ ] Testes de regressão do assembler passam (modo arquivo não é afetado).

## Dependências

- Nenhuma (não depende de outros itens de backlog abertos).

## Risco

- Médio — a integração `BaseLoader` + `FileSystemLoader` para resolução de `extends`/`include` requer um loader composto ou subclasse de `BaseLoader` que delega ao `FileSystemLoader` existente (`DottedLoader`).

## Notas técnicas

- Estratégia sugerida: usar `jinja2.DictLoader({'__stdin__': conteudo})` combinado com `jinja2.ChoiceLoader([DictLoader, DottedLoader(templates_dir)])`.
- O nome interno do template stdin pode ser `'__stdin__'` ou `'-'`.
- `assembler.py` é INTOCÁVEL na Constituição como componente genérico — verificar se a mudança é compatível com Rule 10 (que protege `code/genjin.jinja2`, não `assembler.py`). Confirmar com PO antes de implementar.
