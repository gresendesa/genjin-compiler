# B-025 — Investigação: `get_unhandled_codes` borbulha código já consumido por `LOOP_WHILE` de bloco filho

Status: resolvido (fix em cópia; aplicação ao oficial pendente de aceite do PO)
Prioridade PO: 1 (crítica)

Risco: médio
Data de criação: 2026-05-03

## Descrição do Problema

Na execução do pipeline `compiler.py | assembler.py | indenter.py` com `examples/test.gnj`, o compilador lança a seguinte exceção:

```
Exception: The codes ['DEMORA'] passed to the block 'doe' must be handled locally in loop_while or explicitly passed up using 'pass_codes'.
```

### Fluxo que reproduz o erro

```
exec doe() >> status_var {
    case OK:
        @xpto() when(OK)
        @baz() while(ERROR, DEMORA) when(OK)   ← DEMORA consumido aqui
        @bar()
    pass ERROR, OK
}
```

Após o desugar, a estrutura final produzida para `baz` inclui:
- `LOOP_WHILE: ['DEMORA', 'ERROR']`
- `CASES: [{OUTPUT_CODE: 'OK', BLOCK: bar}]`

O bloco `bar` declara `PASS_CODES: ['DEMORA', 'ERROR', 'OK']`. O código `DEMORA` é **passado para cima por `bar`**, mas deveria ser **absorvido pelo `LOOP_WHILE` de `baz`** antes de chegar ao bloco `doe`.

---

## Causa Suspeita

A macro `get_unhandled_codes_recursively` em `code/genjin.jinja2` coleta recursivamente os códigos não tratados dos blocos filhos. Ao processar o bloco `bar` (filho de `baz`), ela registra `DEMORA` como código não tratado no dicionário `collection_dict` sob a variável correspondente.

No entanto, quando sobe para o nível de `baz`, o filtro que deveria **remover** os códigos presentes em `LOOP_WHILE` de `baz` aparenta não estar sendo aplicado antes de propagar ao nível de `doe`.

O resultado é que `DEMORA` chega até `checks_code_handling` de `doe` como código não tratado, causando a exceção.

---

## Comportamento Esperado (conforme `docs/genjin.md`)

> O `LOOP_WHILE` é avaliado **depois** dos `CASES`. Qualquer código de saída que acione um `LOOP_WHILE` faz o ciclo do **bloco atual reiniciar** — ele não deve ser propagado para o bloco pai.

Portanto, se `DEMORA` está em `LOOP_WHILE` de `baz`, ele é **consumido** naquele nível e **não deve ser borbulhado** para `doe`.

---

## Escopo da Investigação

1. Ler e mapear o fluxo completo de `get_unhandled_codes` → `get_unhandled_codes_recursively` → `checks_code_handling` em `code/genjin.jinja2`.
2. Identificar exatamente onde o filtro por `LOOP_WHILE` falha ao ser aplicado durante a propagação recursiva.
3. Propor correção mínima que:
   - Respeite integralmente a semântica documentada em `docs/genjin.md`.
   - Não quebre nenhum outro cenário de borbulhamento legítimo.
   - Não altere contratos de macro existentes (sem versionamento).
4. Documentar causa raiz e solução em `ia/experience.md` e `ia/decisions.md` se houver decisão de design.
