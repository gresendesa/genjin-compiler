---

## Contexto

O arquivo `code/Federal/@/Lenhador.jinja2` demonstra um padrão manual muito útil: o programador
define `BLOCOS_COMUNS` e `BLOCOS_TRATAMENTO` como dicionários Jinja2 contendo sub-programas
completos, e os reutiliza por referência em múltiplos pontos do programa principal.

A ideia de implementar isso **não é** transpilir para a estrutura de dicionários. Como o processo
pode ser automatizado na transpilação, o compilador simplesmente **expande (inline) o bloco
diretamente** no programa gerado, repetindo a estrutura quantas vezes forem necessárias — sem
intermediário de dicionário.

A motivação principal para parametrizar esses blocos é torná-los **dinâmicos em tempo de
transpilação**: em vez de referenciar variáveis fixas do programa, o proc-bloco recebe parâmetros
que substituem nomes de variáveis ou valores literais no momento em que o bloco é expandido.

---

## Objetivo deste item

**Estudar e propor** como incorporar essa capacidade na linguagem `.gnj` de forma:

1. Coerente com a sintaxe já estabelecida (blocos `vars`, `procs`, `exec`)
2. Intuitiva para o programador
3. Com parâmetros de transpilação que tornam o bloco dinâmico na expansão
4. Segura: **sem recursão direta ou indireta** entre proc-blocos

Este é um item de **pesquisa e proposta**, não de implementação. O entregável é um documento de
decisão registrado em `ia/decisions.md` e uma especificação preliminar em `docs/`.

---

## Conceito central: parâmetros de transpilação

A distinção fundamental desta proposta é que os parâmetros de um proc-bloco **não são parâmetros
de runtime** (não se tornam argumentos de chamada de função no Jinja2 gerado). São
**substituições em tempo de compilação**: quando o transpiler expande o proc-bloco, ele substitui
textualmente os placeholders pelos valores fornecidos na chamada.

**Analogia**: os proc-blocos são macros do compilador — semelhantes a macros C/C++ ou templates
genéricos — mas com a sintaxe familiar do Genjin.

### Exemplo ilustrativo

```gnj
program "lenhador"

procs {
    // Procedimentos normais (runtime)
    // "Federal.@.server_tools.Teleportar" → library: "Federal.@.server_tools", macro: "Teleportar"
    Teleportar(home: &Text) from "Federal.@.server_tools.Teleportar" {
        codes SUCESSO<0>, DEMORA<1>, CANCELADO<2>
    }

    // itens: Text[] — parâmetro plural (lista)
    Guardar_Itens(gaveta: Text, itens: Text[]) from "Federal.@.server_tools.Guardar_Itens" {
        codes SUCESSO<0>, DEMORA<1>
    }

    MinhaRotina() from "Federal.@.Lenhador.minha_rotina" {
        codes SUCESSO<0>, FERRAMENTA_GASTA<1>, OUTRA_FALHA<2>
    }

    // Proc-bloco: não usa "from", não tem chamada de runtime.
    // Parâmetros são placeholders substituídos na transpilação.
    // home_destino: &Text  → placeholder ref (substituído por nome de variável)
    // gaveta_guardar: Text → placeholder lit (substituído por valor literal)
    // REGRA: o exec mais externo NÃO declara >> (herda variável do ponto de chamada)
    Ferramenta_Gasta(home_destino: &Text, gaveta_guardar: Text) {
        exec Teleportar(home=&home_destino) {
            case SUCESSO:
                // NOTA: itens=["diamond_axe"] usa lista literal — gap na spec atual.
                // A sintaxe de lista literal como argumento ainda não está definida
                // em docs/language.md e precisará ser especificada (extensão futura).
                exec Guardar_Itens(gaveta=gaveta_guardar, itens=["diamond_axe"]) {
                    pass SUCESSO, DEMORA
                    // SUCESSO: Guardar_Itens completou → sobe para o exec Teleportar
                    // DEMORA: sobe para o exec Teleportar → capturado por while(DEMORA) → repete
                }
            // Teleportar codes: SUCESSO (→ case), DEMORA (→ while), CANCELADO (→ pass)
            // Guardar_Itens pode retornar SUCESSO (sobe, não capturado → pass) e
            // DEMORA (sobe, capturado por while → repete Teleportar)
            pass CANCELADO, SUCESSO
        } while(DEMORA)
        // Códigos de saída inferidos do proc-bloco: CANCELADO, SUCESSO
    }
}

vars {
    resultado:            Number
    home_machados_gastos: Text
    outra_home:           Text
}

exec MinhaRotina() >> resultado {
    // Chamada 1 — expande o bloco com home_machados_gastos e "slot_a"
    case FERRAMENTA_GASTA: @Ferramenta_Gasta(home_destino=&home_machados_gastos, gaveta_guardar="slot_a")
    // Chamada 2 — expande o mesmo bloco com parâmetros diferentes
    case OUTRA_FALHA:      @Ferramenta_Gasta(home_destino=&outra_home, gaveta_guardar="slot_b")
    // MinhaRotina: SUCESSO (→ pass), FERRAMENTA_GASTA (→ case), OUTRA_FALHA (→ case)
    // Proc-blocos expandidos passam CANCELADO e SUCESSO → ambos precisam estar no pass externo
    pass SUCESSO, CANCELADO
}
```

**O que o transpiler gera para a chamada 1 (exec raiz — após expansão):**

```gnj
exec MinhaRotina() >> resultado {
    case FERRAMENTA_GASTA:
        exec Teleportar(home=&home_machados_gastos) {
            case SUCESSO:
                exec Guardar_Itens(gaveta="slot_a", itens=["diamond_axe"]) {
                    pass SUCESSO, DEMORA
                }
            pass CANCELADO, SUCESSO
        } while(DEMORA)
    case OUTRA_FALHA:
        exec Teleportar(home=&outra_home) {
            case SUCESSO:
                exec Guardar_Itens(gaveta="slot_b", itens=["diamond_axe"]) {
                    pass SUCESSO, DEMORA
                }
            pass CANCELADO, SUCESSO
        } while(DEMORA)
    pass SUCESSO, CANCELADO
}
```

Cada expansão é uma cópia autônoma com os parâmetros substituídos — nenhum dicionário no Jinja2 gerado.

## Distinção entre parâmetro de transpilação e parâmetro de runtime

| Aspecto | Proc normal (`from`) | Proc-bloco (proposta) |
|---|---|---|
| Quando é resolvido | Runtime (Minecraft bot) | Transpilação (compilador Genjin) |
| O que o parâmetro representa | Valor passado ao procedimento externo | Placeholder substituído no bloco |
| Sintaxe de declaração | `param: Type` / `param: &Type` | `param: Type` (lit) / `param: &Type` (ref) — idêntica |
| Sintaxe de chamada | `param=valor` / `param=&var` | `param=valor` / `param=&var` — idêntica |
| Aparece no Jinja2 gerado como | Argumento de função/template call | Valor embutido literalmente no bloco expandido |
| Necessita existir em `vars`? | Sim (se `&`) | Não — é só um nome de substituição |
| Múltiplas chamadas com parâmetros diferentes | Mesmo proc, valores distintos em runtime | Cópias independentes na AST gerada |

## Pontos de design — análise e recomendações (S10-T01)

### 1. Sintaxe de declaração: como distinguir proc-bloco de proc normal

**Opções:**
- A: ausência de `from` + presença de corpo `{ exec ... }` (hipótese PO)
- B: keyword explícita (`template`, `block`, `macro`)

**Análise:** A combinação "sem `from` + abre chave com `exec`" é totalmente não-ambígua na
gramática atual. O parser já faz lookahead para distinguir entre construtos similares; neste caso,
a regra `proc_decl` simplesmente bifurca: se encontra `from` → proc normal; se encontra `{` →
proc-bloco. Isso não cria nenhum token novo e é coerente com outras linguagens que distinguem
"declaração de função" de "definição de função" pelo mesmo critério.

O risco de conflito futuro (querer proc sem `from` para outro fim) é real mas distante e pode ser
resolvido com keyword quando surgir — não justifica uma keyword extra agora.

**Recomendação: Opção A.** Ausência de `from` + corpo `{ exec ... }` identifica o proc-bloco.
Nenhum novo token necessário no scanner.

---

### 2. Declaração dos parâmetros de transpilação

**Proposta:** reutilizar a convenção `&` já existente:
- `param: &Type` → placeholder **ref** — expandido para o nome de uma variável real do programa
- `param: Type` → placeholder **lit** — expandido para um valor literal

**Análise:** A sintaxe é idêntica à declaração de parâmetros de procs normais (`ParamDeclNode`
já possui `evaluation: 'literal' | 'reference'`). Isso significa que o parser não precisará de
nenhuma regra nova para declarar parâmetros de proc-blocos — a regra de `param_decl` existente
já cobre. A diferença semântica (transpilação vs. runtime) é uma propriedade do nó pai
(`ProcBlockNode` vs. `ProcDeclNode`), não do parâmetro em si.

A coerência com a sintaxe de chamada também é total: `=valor` para literal, `=&var` para
referência — exatamente como os `kwargs` de procs normais já funcionam.

**Recomendação: manter `&`/sem `&`.** Sem novo token, sem nova sintaxe, sem novo nó de parâmetro.

---

### 3. Variável de resultado (`>> var`) no corpo do proc-bloco

**Opções:**
- A: variável de resultado como parâmetro de transpilação explícito
- B: bloco mais externo do corpo **não declara `>>`** — herda a variável do ponto de chamada
- C: nome fixo por convenção (ex.: sempre `resultado`)

**Análise:** A linguagem já possui herança de variável (`exec` sem `>>` herda do pai). Um
`@Proc_Bloco()` no corpo de um `case` está sempre dentro de um `exec` que já tem uma variável
ativa. Logo, se o bloco mais externo do proc-bloco não declarar `>>`, ele naturalmente herda a
variável do contexto de expansão — exatamente o comportamento desejado, e sem nenhuma nova
mecânica.

A opção A (parâmetro extra) adicionaria ruído para o caso mais comum (herança simples) e exigiria
uma sintaxe nova para `>> &param_var`, que não existe hoje. A opção C é excessivamente restritiva.

**Recomendação: Opção B.** O bloco mais externo do corp do proc-bloco **não pode declarar `>>`**.
A variável é herdada do ponto de chamada. Blocos internos aninhados podem declarar `>>` normalmente
para variáveis intermediárias. O compilador deve emitir erro se o bloco raiz do proc-bloco tiver `>>`.

**Impacto no exemplo:** corrigir o esboço para remover `>> resultado` do `exec` mais externo.

---

### 4. Inferência de códigos de saída

**Análise:** A linguagem já obriga que todo código de saída seja coberto por `case`, `while` ou
`pass`. Portanto, os `pass_codes` do bloco mais externo de um proc-bloco são **determinísticos e
completos por construção** — não há ambiguidade possível. O compilador (parser + desugar) já
valida isso para `exec` normais; a mesma regra se aplica ao corpo do proc-bloco.

Declaração explícita de `codes` seria redundante e criaria oportunidade de erro (declarar codes
que não batem com os `pass` reais).

**Recomendação: inferência automática obrigatória.** Os códigos de saída do proc-bloco são
exatamente os `pass_codes` do bloco raiz interno. Nenhuma sintaxe `codes { }` no proc-bloco.

---

### 5. Detecção de recursão

**Análise:** Um proc-bloco pode invocar outros proc-blocos via `@Outro_Bloco()`. O compilador
precisa garantir ausência de ciclos no grafo de dependência antes de fazer qualquer expansão —
pois a expansão de um ciclo seria infinita.

A fase correta para essa verificação é o **início de `desugar.py`**, antes de qualquer expansão:
1. Construir o mapa de todos os `ProcBlockNode` do programa
2. Para cada proc-bloco, coletar as chamadas `@Proc()` cujo alvo é outro proc-bloco
3. Fazer DFS/coloração sobre o grafo resultante
4. Ciclo detectado → `DesugarError` com mensagem listando o caminho do ciclo

Isso se encaixa diretamente no início de `desugar()`, junto ao `proc_map` que já é construído.
Não justifica uma fase separada.

**Recomendação: validação no início de `desugar.py`** com DFS sobre grafo de dependência entre
proc-blocos. A ordem topológica resultante também determina a ordem de expansão (proc-blocos sem
dependências são expandidos primeiro).

---

### 6. Fase de expansão: `desugar.py` vs. `transpiler.py`

**Análise:** `desugar.py` já é a fase de transformação de AST — seu contrato é: recebe AST com
nós sintáticos especiais e devolve AST canônica. Expandir proc-blocos é exatamente isso: um nó
de chamada (`@Proc_Bloco(params)`) é substituído pela AST do bloco interno com parâmetros
aplicados. O `transpiler.py` só deve receber AST canônica — misturar lógica de expansão nele
violaria essa separação.

**Recomendação: `desugar.py`.** A expansão de `@Proc_Bloco(params)` acontece inteiramente no
desugar, na mesma passagem que desaçucara `@proc()` (B-015). O desugar precisará distinguir os
dois casos: se o nome do `@` aponta para um `ProcBlockNode` → expansão com clonagem e substituição;
se aponta para `ProcDeclNode` → expansão atual (B-015).

---

### 7. Viabilidade técnica da substituição de parâmetros

**Análise:** Ao expandir `@Ferramenta_Gasta(home_destino=&home_machados_gastos, gaveta_guardar="slot_a")`,
o desugar precisa:

1. Localizar o `ProcBlockNode` de `Ferramenta_Gasta`
2. Construir um mapeamento `{placeholder_name → ArgNode}` a partir dos kwargs da chamada
3. Fazer **deep clone** da AST do bloco (para não poluir a definição original — que pode ser
   expandida novamente com outros parâmetros)
4. Percorrer o clone com um **visitor de substituição** que, em cada `ArgNode` cujo `value` bate
   com um placeholder, substitui pelo `ArgNode` do mapeamento

Python oferece `copy.deepcopy` para o clone. O visitor é uma função recursiva que percorre
`ExecBlockNode → CaseNode → ExecBlockNode` (já existe esse padrão em `_desugar_exec`).

A substituição em si é simples: `ArgNode.value` é `str` para referência. Se o valor for igual ao
nome de um placeholder `&`-ref, substitui pelo `value` do ArgNode correspondente no mapeamento.
Para literais, substitui o `value` direto.

Nenhuma nova estrutura de dado é necessária. A lógica cabe em ~30 linhas no desugar.

**Recomendação: Opção B (deep clone + visitor).** Implementação direta com as estruturas
existentes. A AST original do proc-bloco permanece imutável e pode ser expandida múltiplas vezes.

**Veredicto: zero mudanças no scanner para B-017. Mudança de scanner fica em B-018.**

---

### `compiler/parser.py` — maior impacto (risco alto)

É a fase de maior complexidade. Mudanças necessárias:

#### 1. Novo nó AST: `ProcBlockNode`

```python
@dataclass
class ProcBlockNode:
    name: str
    parameters: list[ParamDeclNode]   # placeholders de transpilação
    block: ExecBlockNode | InlineSeqNode
```

#### 2. Tipo de `ProgramNode.procedures` muda

```python
# antes
procedures: list[ProcDeclNode]
# depois
procedures: list[ProcDeclNode | ProcBlockNode]
```

#### 3. `_parse_procs` — dois passagens para proc-blocos

O parser atual usa uma abordagem de dois passos para o `exec` raiz (skip na primeira passagem,
parse na segunda com `declared_procs` completo). O mesmo padrão é necessário para os corpos de
proc-blocos: um proc-bloco pode referenciar um proc declarado **depois** dele em `procs { }`.

Estratégia: durante `_parse_procs`, ao encontrar um proc-bloco (sem `from`), armazenar a posição
do token e parsear o corpo em um segundo passo, após todo o bloco `procs { }` ser lido.

#### 4. `_parse_proc_decl` — bifurcação no `from`

```python
def _parse_proc_decl(self) -> ProcDeclNode | ProcBlockNode:
    name_tok = ...
    self._expect(LPAREN)
    params = self._parse_param_list()
    self._expect(RPAREN)
    if self._check(KW_FROM):
        # proc normal — caminho atual
        ...
        return ProcDeclNode(...)
    elif self._check(LBRACE):
        # proc-bloco — novo caminho
        ...
        return ProcBlockNode(...)
    else:
        raise ParseError("esperado 'from' ou '{'", ...)
```

#### 5. Corpo do proc-bloco: `declared_vars` aumentado

O corpo do proc-bloco usa parâmetros como pseudo-variáveis. Ao parsear o corpo, passar:

```python
pseudo_vars = declared_vars | {p.name for p in params if p.evaluation == 'reference'}
```

Isso permite que `&home_destino` passe a validação de referência de variável sem estar em `vars`.

#### 6. Chamadas `@ProcBloco()` e `exec ProcBloco()`: validação diferida

O parser atual, ao ver `@proc()`, valida os códigos do `when/while/case` contra
`proc_decl.output_codes`. Para proc-blocos, os códigos de saída são inferidos do corpo (não
declarados). Duas opções:
- A: inferir `output_codes` do `ProcBlockNode` durante o parse do corpo (percorrer `pass_codes`
  do bloco raiz) — permite validação imediata
- B: diferir toda validação de códigos de proc-blocos para o desugar

**Recomendação: Opção A.** Após parsear o corpo do proc-bloco, percorrer o bloco raiz e coletar
os `pass_codes` como `output_codes` sintéticos — armazenar em `ProcBlockNode.inferred_codes`.
Isso mantém a validação no parser e não sobrecarrega o desugar.

**Estimativa: ~120-150 linhas de mudança/adição no `parser.py`.**

---

### `compiler/desugar.py` — segundo maior impacto (risco médio)

#### 1. Detecção de recursão

No início de `desugar()`, após construir o `proc_map`, construir `block_map` (só `ProcBlockNode`)
e fazer DFS para detectar ciclos. ~30 linhas.

#### 2. `proc_map` precisa incluir proc-blocos

`proc_map` atualmente é `dict[str, ProcDeclNode]`. Passa a ser
`dict[str, ProcDeclNode | ProcBlockNode]`. Isso afeta `_desugar_seq` e `_build_terminal`/
`_build_chained` que consultam `proc_map`.

#### 3. Expansão de `@ProcBloco()` e `exec ProcBloco()`

Nos builders `_build_terminal` e `_build_chained`, quando o alvo é `ProcBlockNode`:
1. Construir mapeamento `{placeholder → ArgNode}` dos kwargs da chamada
2. `copy.deepcopy` da AST do `ProcBlockNode.block`
3. Aplicar visitor de substituição sobre o clone
4. Retornar o clone já desaçucarado recursivamente

Visitor de substituição percorre `ExecBlockNode → CaseNode → ExecBlockNode` substituindo
`ArgNode.value` onde bate com um placeholder. ~40 linhas.

#### 4. Ordem de expansão

A DFS do item 1 produz ordem topológica. Expandir proc-blocos folha primeiro (sem dependências
em outros proc-blocos), avançando até a raiz. Isso garante que ao expandir `A → B`, `B` já
esteja completamente expandido.

**Estimativa: ~80-100 linhas de mudança/adição no `desugar.py`.**

---

### `compiler/transpiler.py` — mínimo (risco baixo)

Se o desugar expandir completamente todos os proc-blocos antes da transpilação, o transpiler
recebe apenas `ExecBlockNode` canônicos — **nenhuma mudança necessária**.

A única atenção: `_write_proc` itera sobre `ast.procedures` e espera `ProcDeclNode`. Após a
expansão, `ProcBlockNode` deve ser **removido de `procedures`** (ou filtrado pelo transpiler).

Recomendação: o desugar já retorna um `ProgramNode` novo (sem modificar o original). Basta que
o novo `ProgramNode.procedures` contenha apenas `ProcDeclNode` (proc-blocos filtrados fora).

**Estimativa: 0 linhas de mudança no transpiler. Filtro de proc-blocos fica em `desugar.py`.**

---

### `compiler/ast_io.py` — médio (risco baixo)

Precisa serializar/desserializar `ProcBlockNode`. O padrão já existe para os outros nós — é uma
extensão direta de `_node_to_dict` e `_dict_to_node`.

**Estimativa: ~30 linhas.**

---

### Fase de validação semântica — embutida no parser e no desugar

Não justifica fase própria. Distribuição:
- Parser: validação de tipos (`Object` proibido em `vars`, etc.) — B-018
- Parser: `declared_vars` aumentado para aceitar parâmetros como pseudo-vars
- Desugar início: detecção de recursão

## Risco

Médio-alto. Extensão semântica relevante: parâmetros de transpilação são um conceito novo na
linguagem que exige cuidado para não criar ambiguidade com parâmetros de runtime. O principal
risco é a fase de substituição de parâmetros na AST — se mal implementada, pode gerar Jinja2
incorreto silenciosamente. Mitigado por fazer este item ser apenas pesquisa/proposta antes de
qualquer implementação, e por cobrir detecção de recursão como requisito obrigatório.
