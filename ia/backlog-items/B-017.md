## Contexto

O arquivo `code/Federal/@/Lenhador.jinja2` demonstra um padrão manual muito útil: o programador
define `BLOCOS_COMUNS` e `BLOCOS_TRATAMENTO` como dicionários Jinja2 contendo sub-programas
completos, e os reutiliza por referência em múltiplos pontos do programa principal.

A ideia de implementar isso **não é** transpilir para a estrutura de dicionários. Como o processo
pode ser automatizado na transpilação, o compilador simplesmente **expande (inline) o bloco
diretamente** no programa gerado, repetindo a estrutura quantas vezes forem necessárias — sem
intermediário de dicionário.

A motivação principal para parametrizar esses blocos é torná-los **dinâmicos em tempo de
transpilação**: em vez de referenciar variáveis fixas do programa, o proc-bloco recebe parâmetros
que substituem nomes de variáveis ou valores literais no momento em que o bloco é expandido.

---

## Objetivo deste item

**Estudar e propor** como incorporar essa capacidade na linguagem `.gnj` de forma:

1. Coerente com a sintaxe já estabelecida (blocos `vars`, `procs`, `exec`)
2. Intuitiva para o programador
3. Com parâmetros de transpilação que tornam o bloco dinâmico na expansão
4. Segura: **sem recursão direta ou indireta** entre proc-blocos

Este é um item de **pesquisa e proposta**, não de implementação. O entregável é um documento de
decisão registrado em `ia/decisions.md` e uma especificação preliminar em `docs/`.

---

## Conceito central: parâmetros de transpilação

A distinção fundamental desta proposta é que os parâmetros de um proc-bloco **não são parâmetros
de runtime** (não se tornam argumentos de chamada de função no Jinja2 gerado). São
**substituições em tempo de compilação**: quando o transpiler expande o proc-bloco, ele substitui
textualmente os placeholders pelos valores fornecidos na chamada.

**Analogia**: os proc-blocos são macros do compilador — semelhantes a macros C/C++ ou templates
genéricos — mas com a sintaxe familiar do Genjin.

### Exemplo ilustrativo

```gnj
program "lenhador"

procs {
    // Procedimentos normais (runtime)
    // "Federal.@.server_tools.Teleportar" → library: "Federal.@.server_tools", macro: "Teleportar"
    Teleportar(home: &Text) from "Federal.@.server_tools.Teleportar" {
        codes SUCESSO<0>, DEMORA<1>, CANCELADO<2>
    }

    Guardar_Itens(gaveta: Text, itens: Text[]) from "Federal.@.server_tools.Guardar_Itens" {
        codes SUCESSO<0>, DEMORA<1>
    }

    MinhaRotina() from "Federal.@.Lenhador.minha_rotina" {
        codes SUCESSO<0>, FERRAMENTA_GASTA<1>, OUTRA_FALHA<2>
    }

    // Proc-bloco: não usa "from", não tem chamada de runtime.
    // Parâmetros são placeholders substituídos na transpilação.
    // home_destino: &Text  → placeholder do tipo ref (será substituído por nome de variável)
    // gaveta_guardar: Text → placeholder do tipo lit (será substituído por valor literal)
    Ferramenta_Gasta(home_destino: &Text, gaveta_guardar: Text) {
        exec Teleportar(home=&home_destino) >> resultado {
            case SUCESSO:
                exec Guardar_Itens(gaveta=gaveta_guardar, itens=["diamond_axe"]) {
                    pass SUCESSO, DEMORA
                }
            pass CANCELADO
        } while(DEMORA)
    }
}

vars {
    resultado:            Number
    home_machados_gastos: Text
    outra_home:           Text
}

exec MinhaRotina() >> resultado {
    // Chamada 1 — expande o bloco com home_machados_gastos e "slot_a"
    case FERRAMENTA_GASTA: @Ferramenta_Gasta(home_destino=&home_machados_gastos, gaveta_guardar="slot_a")
    // Chamada 2 — expande o mesmo bloco com parâmetros diferentes
    case OUTRA_FALHA:      @Ferramenta_Gasta(home_destino=&outra_home, gaveta_guardar="slot_b")
    pass SUCESSO
}
```

**O que o transpiler gera para a chamada 1:**

O bloco `Ferramenta_Gasta` é **inlineado** na estrutura do programa Jinja2, com `home_destino`
substituído pelo nome de variável `home_machados_gastos` (ref) e `gaveta_guardar` substituído
pelo valor literal `"slot_a"`. Para a chamada 2, o mesmo bloco é inlineado novamente com os
outros valores — duas cópias independentes na árvore de execução gerada.

Isso é diferente de dicionários: não há referência compartilhada — cada expansão é autônoma.

---

## Distinção entre parâmetro de transpilação e parâmetro de runtime

| Aspecto | Proc normal (`from`) | Proc-bloco (proposta) |
|---|---|---|
| Quando é resolvido | Runtime (Minecraft bot) | Transpilação (compilador Genjin) |
| O que o parâmetro representa | Valor passado ao procedimento externo | Placeholder substituído no bloco |
| Sintaxe de declaração | `param: Type` / `param: &Type` | `param: Type` (lit) / `param: &Type` (ref) — idêntica |
| Sintaxe de chamada | `param=valor` / `param=&var` | `param=valor` / `param=&var` — idêntica |
| Aparece no Jinja2 gerado como | Argumento de função/template call | Valor embutido literalmente no bloco expandido |
| Necessita existir em `vars`? | Sim (se `&`) | Não — é só um nome de substituição |
| Múltiplas chamadas com parâmetros diferentes | Mesmo proc, valores distintos em runtime | Cópias independentes na AST gerada |

## Pontos de design a investigar

### 1. Sintaxe de declaração: como distinguir proc-bloco de proc normal

- **Ausência de `from`** identifica o proc-bloco (hipótese PO — mais limpa)
- Risco: conflito se futuramente quisermos proc sem `from` para outro fim
- Alternativa: keyword explícita (`template`, `block`, `macro`)
- **A avaliar**: a ausência de `from` + presença de `{ exec ... }` é suficientemente não-ambígua?

### 2. Declaração dos parâmetros de transpilação

A proposta é reutilizar a convenção `&` já existente na linguagem para distinguir os dois tipos
de placeholder:

- `param: &Text` → placeholder do tipo **ref**: na expansão, o compilador substitui `param` pelo
  nome de uma variável real do programa (passada com `=&var` na chamada)
- `param: Text` → placeholder do tipo **lit**: na expansão, o compilador substitui `param` pelo
  valor literal (passado com `=valor` na chamada)

Isso mantém total coerência com a declaração de parâmetros de procs normais e com a sintaxe de
argumentos de chamada já existente (`=valor` para literal, `=&var` para referência).

**A avaliar**: a distinção `&` / sem `&` é suficientemente clara para o programador entender que
esses parâmetros são resolvidos em tempo de compilação, não em runtime?

### 3. Variável de resultado (`>> var`)

- O bloco interno usa `>> resultado` com um nome de variável real do programa
- Opção A: a variável de resultado **também é um parâmetro de transpilação** — `Ferramenta_Gasta(home_destino: ref, resultado_var: ref)` — máxima flexibilidade
- Opção B: o bloco herda a variável do `exec` pai no ponto de chamada — mais simples mas menos explícito
- Opção C: variável de resultado é sempre `resultado` (fixa por convenção) — mais restritivo
- **A avaliar**: qual balanceia expressividade e simplicidade?

### 4. Inferência de códigos de saída

- Os `pass` do bloco interno determinam os códigos que saem do proc-bloco
- O transpiler pode **inferir** automaticamente percorrendo a AST do bloco
- Alternativa: o programador declara explicitamente `codes { SUCESSO, CANCELADO }` após a chave de fechamento
- **A avaliar**: inferência automática é suficientemente determinística? Há casos em que a inferência seria ambígua?

### 5. Detecção de recursão

- O compilador constrói um **grafo de dependência** entre proc-blocos (quais proc-blocos são referenciados dentro de cada um)
- Ciclos no grafo → erro de compilação com mensagem indicando o caminho do ciclo
- Cobertura obrigatória: recursão **indireta** (A → B → A)
- Proc-blocos podem referenciar outros proc-blocos (composição permitida), desde que sem ciclos
- **A avaliar**: essa verificação pode ser feita no parser/desugar ou precisa de fase própria?

### 6. Expansão no transpiler vs. no desugar

A expansão de `@Proc_Bloco(params)` é fundamentalmente diferente da expansão atual de `@proc()` (B-015):

- B-015: `@proc()` → envolve o proc num `exec` canônico (estrutura simples)
- B-017: `@Proc_Bloco(params)` → **substitui** a chamada pela AST completa do bloco interno, com parâmetros aplicados

Questão de fase:
- Fazer no `desugar.py` (pré-AST, antes do transpiler): mais simples, trabalha na AST já existente
- Fazer no `transpiler.py` (geração Jinja2): mais controle, mas mistura responsabilidades
- **Recomendação a avaliar**: desugar parece o lugar certo (já lida com transformações de AST)

### 7. Viabilidade técnica da substituição de parâmetros

O desafio central: o bloco interno usa nomes como `home_destino` que **não existem em `vars`**.
O transpiler precisa saber, ao percorrer a AST do bloco, que `home_destino` é um placeholder e não
uma variável real. Opções:

- A: a AST do proc-bloco carrega os parâmetros anotados; o desugar faz substituição antes de inserir o bloco no ponto de chamada
- B: o desugar cria uma cópia da AST do bloco e aplica um visitor de substituição sobre ela

Ambas são viáveis tecnicamente. A opção B é mais robusta (não polui a AST original).

## Risco

Médio-alto. Extensão semântica relevante: parâmetros de transpilação são um conceito novo na
linguagem que exige cuidado para não criar ambiguidade com parâmetros de runtime. O principal
risco é a fase de substituição de parâmetros na AST — se mal implementada, pode gerar Jinja2
incorreto silenciosamente. Mitigado por fazer este item ser apenas pesquisa/proposta antes de
qualquer implementação, e por cobrir detecção de recursão como requisito obrigatório.
